# Phase 4 — status and pickup guide

Authoritative pickup guide for **Phase 4: Device model + userspace drivers** (per `ROADMAP.md §6`).

## TL;DR

Phase 4 lifts the kernel from "userspace runs" (Phase 3 close) to **"hardware is reachable from userspace via Plan 9 9P"**. Major deliverables: Dev vtable (Plan 9 idiom verbatim) + Spoor lifecycle infrastructure; kernel-internal synthetic Devs (`/dev/cons`, `/dev/null`, `/dev/zero`, `/dev/random`, `/proc`, `/ctl`, `/ramfs`); VirtIO core (in-kernel transport, MMIO + minimal PCIe); IRQ forwarding to userspace via `KObj_IRQ` blocker wakeups; userspace Rust drivers for virtio-blk / virtio-net / virtio-input / virtio-gpu; driver crash supervision.

The phase culminates with userspace drivers exposing their devices as 9P servers at `/dev/<name>/`. Phase 5 (= ROADMAP §7 by local numbering) then layers 9P client + Stratum integration on top.

Per CLAUDE.md spec-first policy: VirtIO core is impl-only (transport correctness checked by integration tests + adversarial audit; no new TLA+). BURROW refcount + mapping lifecycle (`specs/burrow.tla`) is finalized this phase. IRQ-handle wakeup is covered by `scheduler.tla`'s wait/wake protocol (already proved I-9 NoMissedWakeup).

**Local phase numbering note**: this is "Phase 4" by local convention; ROADMAP §6 is the canonical scripture text (titled "Phase 3" in the ROADMAP but representing the device-model phase that comes after our local Phase 3 close). The local renumbering stems from splitting ROADMAP Phase 2's deferred address-space deliverables across local Phase 2 + Phase 3. Working name: **P4-* sub-chunks**.

## Trip-hazard #157 — CLOSED at P4-Fix157

**Status (2026-05-07)**: **CLOSED.** Root-caused to `userland_enter` SPSel discipline, not a QEMU virt simulation artifact. Fix: `msr SPSel, #0; isb` immediately before `mov sp, x17` in `arch/arm64/userland.S`. Regression test in `kernel/test/test_userspace2.c` (`userspace.first_iteration` + `userspace.second_iteration`).

See `memory/audit_r7_closed_list.md` § "Trip-hazard #157 — CLOSED at P4-Fix157" for the full root-cause analysis, ruled-out hypotheses, and fix discussion. Quick summary:

- `userland_enter`'s `mov sp, x17` writes SP_EL0 only when SPSel=0. Under the IRQ-driven dispatch path the kthread runs at SPSel=1, so `mov sp` clobbers SP_EL1 instead. After eret to EL0, the user code's first SVC enters EL1 with the corrupt SP_EL1 (= user_sp = 0x80000000, low VA, unmapped); KERNEL_ENTRY's `sub sp + stp` loops infinitely with FAR decreasing by EXCEPTION_CTX_SIZE = 0x120 per cycle.
- The R7 hypothesis chain (MTE / TLB / walker cache / PA recycle) was misdirected — verified by ablation. State was "bit-identical" only because the audit didn't print PSTATE.SPSel or SP_EL1.
- The fix is one `msr SPSel, #0; isb` before `mov sp, x17`. Iter 1 + iter 2 both pass after the fix. P4-I (virtio-blk) and P4-M (driver supervision) are unblocked.

183. **All userspace SPIs are edge-triggered at v1.0 (P4-Ic5b2 GIC ICFGR transition).** `kobj_irq_create` flips `GICD_ICFGR` to 0b10 (edge) for SPIs ≥ 32 before `gic_enable_irq`. QEMU virt's virtio-mmio + most ARM device IRQs are edge per DTB. A future level-triggered consumer (PCIe legacy INTx) would need a flag on `t_irq_create`. Setting edge on a level-driven line wedges silently; setting level on an edge-driven line either fires infinitely (level-after-edge) or fails (edge detector unarmed for non-LO_HI signalling). **Tracked as R12-gic-edge deferred audit.**

184. **`gic_set_spi_edge_triggered` RMW is not SMP-safe for concurrent claims of different SPIs in the same 32-bit ICFGR register.** Two CPUs claiming different SPIs in the same 16-INTID window could lose updates. At v1.0 IRQ claims happen on a single thread (kproc + serial test runner). Phase 5+ should add a lock or use load-link/store-conditional. Documented in `39-hw-handles.md` footgun #8.
