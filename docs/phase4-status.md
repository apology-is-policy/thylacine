# Phase 4 — status and pickup guide

Authoritative pickup guide for **Phase 4: Device model + userspace drivers** (per `ROADMAP.md §6`).

## TL;DR

Phase 4 lifts the kernel from "userspace runs" (Phase 3 close) to **"hardware is reachable from userspace via Plan 9 9P"**. Major deliverables: Dev vtable (Plan 9 idiom verbatim) + Spoor lifecycle infrastructure; kernel-internal synthetic Devs (`/dev/cons`, `/dev/null`, `/dev/zero`, `/dev/random`, `/proc`, `/ctl`, `/ramfs`); VirtIO core (in-kernel transport, MMIO + minimal PCIe); IRQ forwarding to userspace via `KObj_IRQ` blocker wakeups; userspace Rust drivers for virtio-blk / virtio-net / virtio-input / virtio-gpu; driver crash supervision.

The phase culminates with userspace drivers exposing their devices as 9P servers at `/dev/<name>/`. Phase 5 (= ROADMAP §7 by local numbering) then layers 9P client + Stratum integration on top.

Per CLAUDE.md spec-first policy: VirtIO core is impl-only (transport correctness checked by integration tests + adversarial audit; no new TLA+). BURROW refcount + mapping lifecycle (`specs/burrow.tla`) is finalized this phase. IRQ-handle wakeup is covered by `scheduler.tla`'s wait/wake protocol (already proved I-9 NoMissedWakeup).

**Local phase numbering note**: this is "Phase 4" by local convention; ROADMAP §6 is the canonical scripture text (titled "Phase 3" in the ROADMAP but representing the device-model phase that comes after our local Phase 3 close). The local renumbering stems from splitting ROADMAP Phase 2's deferred address-space deliverables across local Phase 2 + Phase 3. Working name: **P4-* sub-chunks**.

### NEW at P4-Ic5b2

181. **~~Scheduler "no runnable peer system-wide" deadlock check is too aggressive for hardware-IRQ waiters (R12-sched).~~ CLOSED at P4-Ic6-impl.** Two-layer close: P4-Ic6-spec extended scheduler.tla with `HardwareWakeProgress` + `Liveness_Hwake` fairness (SF on Wake encoding "hardware IRQs eventually deliver"). Test workaround retired: `test_virtio_blk_probe.c` now uses direct `wait_pid` instead of the yield-poll loop. See P4-Ic6-spec + P4-Ic6-impl landed-chunk rows.

182. **QEMU virt's virtio-mmio bus is legacy by default; modern transport requires `-global virtio-mmio.force-legacy=false`** (P4-Ic5b2). Without the flag, `VIRTIO_MMIO_VERSION` reads 1 (legacy) instead of 2 (modern). The Thylacine virtio-blk driver crate targets the modern transport (legacy uses a different MMIO register layout with `GuestPageSize` / `QueuePFN` semantics that we don't support). `tools/run-vm.sh` sets the global. Real-hardware ARM virtio-mmio typically already provides the modern transport per VIRTIO 1.2.

183. **All userspace SPIs are edge-triggered at v1.0 (P4-Ic5b2 GIC ICFGR transition).** `kobj_irq_create` flips `GICD_ICFGR` to 0b10 (edge) for SPIs ≥ 32 before `gic_enable_irq`. QEMU virt's virtio-mmio + most ARM device IRQs are edge per DTB. A future level-triggered consumer (PCIe legacy INTx) would need a flag on `t_irq_create`. Setting edge on a level-driven line wedges silently; setting level on an edge-driven line either fires infinitely (level-after-edge) or fails (edge detector unarmed for non-LO_HI signalling). **Tracked as R12-gic-edge deferred audit.**

184. **`gic_set_spi_edge_triggered` RMW is not SMP-safe for concurrent claims of different SPIs in the same 32-bit ICFGR register.** Two CPUs claiming different SPIs in the same 16-INTID window could lose updates. At v1.0 IRQ claims happen on a single thread (kproc + serial test runner). Phase 5+ should add a lock or use load-link/store-conditional. Documented in `39-hw-handles.md` footgun #8.

### NEW at P4-Ic6-spec

185. **~~R12-sched impl deferral: `on_cpu` protocol intersection with WFI-in-place is architectural.~~ CLOSED at P4-Ic6-impl.** Direction B (separate boot-CPU idle thread distinct from kthread) implemented. `g_bootcpu_idle` allocated via `thread_create(kproc(), bootcpu_idle_main)` in boot_main + registered via `sched_set_bootcpu_idle()` + kept OFF the run tree (to avoid preempt-time SpSel=1 context switches that would clobber SP_EL1 = exception stack). sched()'s deadlock path explicitly switches to `g_bootcpu_idle` (only from VOLUNTARY callers at SpSel=0; the on_cpu spin in wakeup() works correctly because cpu_switch_context writes SP_EL0 at SpSel=0 and the prev's on_cpu is cleared via the standard P2-Cf handoff). Plan 9 heritage alignment: bootcpu_idle is the natural "no work" idle state, not an error state. Closes R12-sched + R12-sched-impl deferred audits. Direction A (refactor on_cpu with THREAD_HALTED sub-state) NOT taken — would have introduced a new sub-state with subtle cross-CPU rules; Direction B reuses existing P2-Cf mechanism unchanged. See P4-Ic6-impl landed-chunk row.

186. **~~Pre-existing: `scheduler_liveness_wfi.cfg` violates LatencyBound (since P3-G landing at `7a61651`)~~ CLOSED at P4-Ic6-cfg.** The "cfg parameters too tight" hypothesis was wrong. Fix at P4-Ic6-cfg: Wake/WakeAll atomically clear wfi at their CHOOSE'd target (Fix A); cfg reparameterized 2T × 2C → 1T × 2C. The original F77/F78 verification intent (negative case: WFI starvation without IPI fairness) is preserved by the still-2T × 2C `scheduler_buggy_wfi.cfg`.
