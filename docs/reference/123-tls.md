# 123 -- `usr/lib/tls`: native TLS for libthyla-rs clients

## Purpose

`usr/lib/tls` is the native-userspace TLS substrate (NET-DESIGN section 9). It
wraps **rustls** (pure-Rust TLS) with the **RustCrypto** `CryptoProvider`,
no_std, on `aarch64-unknown-none`. TLS is a userspace library concern: netd
serves raw TCP `data` over `/net`, and the handshake runs in the client over
that byte stream -- the kernel and netd never see TLS. This keeps the cert-trust
policy in userspace where it belongs.

As of net-7c-1 the crate proves feasibility (the full crypto path runs in-guest)
and exposes the client config + a ClientHello driver. The `TlsStream` transport
(the unbuffered state machine over a `/net` `TcpStream`), the PEM root loader,
and the https-fetch tool land at net-7c-2 with their loopback handshake E2E.

## Public API

```rust
// Build a TLS 1.3 client config: the RustCrypto provider, the LS-K wall clock,
// and `roots` as trust anchors (the baked CA bundle).
pub fn client_config(roots: rustls::RootCertStore) -> Arc<rustls::ClientConfig>;

// Drive the handshake to its first flight and return the ClientHello record.
// Proves the provider + CSPRNG + ECDHE key share + record layer run on the
// target. Returns None on any state-machine error.
pub fn client_hello(host: &str) -> Option<alloc::vec::Vec<u8>>;
```

## Implementation

The crate is `#![no_std]` + `extern crate alloc`. Three pieces make rustls run
on bare metal:

1. **Custom getrandom backend.** `register_custom_getrandom!(thyla_rng)` routes
   `getrandom` to the kernel CSPRNG via `libthyla_rs::t_getrandom` (`SYS_GETRANDOM`).
   The RustCrypto provider draws the client random + the ECDHE ephemeral key
   share through getrandom, which has no `aarch64-unknown-none` backend. The
   registration lives here (the only getrandom user in native userspace), so
   every `tls` consumer inherits it; the macro defines `__getrandom_custom`,
   linked only into binaries that depend on `tls`.

2. **`TimeProvider` over LS-K `CLOCK_REALTIME`.** no_std rustls has no built-in
   clock; webpki reads the wall time to bound a server cert's validity window.
   `ThylaTime::current_time` returns `UnixTime::since_unix_epoch(SystemTime::now().since_epoch())`.
   This is the structural reason TLS follows net-7a (the clock work): a native
   TLS client cannot validate a certificate without a real wall clock.

3. **`builder_with_details(provider, time_provider)`.** The no_std entry point;
   the std `ClientConfig::builder()` (which supplies a default provider + clock)
   is `std`-gated. The connection type is likewise the no_std
   `UnbufferedClientConnection` (the std `ClientConnection` wraps `std::io`).

`client_hello` constructs an `UnbufferedClientConnection`, calls
`process_tls_records(&mut [])` (no incoming bytes yet), and on the resulting
`ConnectionState::EncodeTlsData` calls `encode(&mut out)` -- yielding the
ClientHello TLS record (`out[0] == 0x16`, the handshake content type).

### Cargo recipe (the load-bearing feature flags)

```toml
rustls = { version = "0.23", default-features = false, features = ["custom-provider"] }
rustls-rustcrypto = { version = "0.0.2-alpha", default-features = false, features = ["alloc", "zeroize"] }
rustls-pki-types = { version = "1", default-features = false, features = ["alloc"] }
getrandom = { version = "0.2", features = ["custom"] }
```

- `custom-provider` drops the built-in aws-lc-rs/ring providers (both compile
  C/asm and do not build bare-metal).
- `rustls-rustcrypto` `default-features = false` drops its `std` (which cascades
  `getrandom/std`, breaking the no_std build with `extern crate std`).
- **`tls12` is intentionally omitted** -> TLS 1.3-only (size; see below).

## Constraints (net-7c-1 findings)

- **`CAP_CSPRNG_READ` is mandatory.** A `tls` client must be spawned WITH
  `CAP_CSPRNG_READ` (a plain `t_spawn` confers no caps). The spawning authority
  (warden / login / joey) grants it, exactly as joey grants it to corvus.
  Without it `t_getrandom` returns -1, the key share cannot be generated, and
  the handshake fails closed.

- **Binary size at the 1 MiB `SYS_SPAWN_BLOB_MAX` ceiling.** The TLS-1.3 smoke
  is ~1,006,400 bytes (with TLS 1.2 + RSA-kx it is ~1,067,432, OVER the cap; the
  kernel slurps a spawned ELF into a blob bounded by `SYS_SPAWN_BLOB_MAX`).
  v1.0 native TLS is **TLS 1.3-only**. The net-7c-2 size strategy is to strip
  the TLS binaries (no userspace tool consumes their symbols; keep an unstripped
  copy under `build/` for manual fault-PC resolution) -- pure build config, no
  kernel change; a cap bump is the fallback.

## Status

- **net-7c-1 (landed):** the `tls` crate (provider + getrandom backend + LS-K
  TimeProvider + `client_config` + `client_hello`) + `usr/tls-smoke` (the
  in-guest runtime prover: emits a real ClientHello, asserts it is a valid
  handshake record) + the joey boot probe (spawned with `CAP_CSPRNG_READ`).
  Proven: compile + link + size (1.3-only fits) + runtime (the RustCrypto
  provider, the kernel CSPRNG, and the record layer all execute on aarch64,
  in-guest). Kernel byte-unchanged.
- **net-7c-2 (next):** the `TlsStream` transport (the unbuffered state machine
  over a `/net` `TcpStream`) + the PEM root loader over the baked
  `/etc/ssl/certs/ca-certificates.crt` + an `https` GET tool + the deterministic
  in-memory loopback handshake E2E (client <-> server, baked test cert) + the
  size strategy (strip) + a best-effort real-https probe.
- **net-7d:** the focused audit of the TLS cert-validation surface + the SMP
  gate + close.

## Tests

- `usr/tls-smoke` (`/bin/tls-smoke`): the in-guest runtime proof, gated on every
  boot (joey reaps it + asserts exit 0 + the `tls-smoke: ClientHello OK` marker).
- The loopback handshake E2E (client <-> server over an isolated in-memory pipe)
  lands at net-7c-2 -- the deterministic, peer-independent proof that a full
  handshake completes in-guest.

## Naming rationale

The crate keeps the standard name `tls` -- the reader expects it, and a thematic
rename would obscure intent (the bar from CLAUDE.md "Thematic naming").
