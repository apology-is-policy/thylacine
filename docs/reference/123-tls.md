# 123 -- `usr/lib/tls`: native TLS for libthyla-rs clients

## Purpose

`usr/lib/tls` is the native-userspace TLS substrate (NET-DESIGN section 9). It
wraps **rustls** (pure-Rust TLS) with the **RustCrypto** `CryptoProvider`,
no_std, on `aarch64-unknown-none`. TLS is a userspace library concern: netd
serves raw TCP `data` over `/net`, and the handshake runs in the client over
that byte stream -- the kernel and netd never see TLS. This keeps the cert-trust
policy in userspace where it belongs.

net-7c-1 proved feasibility (the full crypto path runs in-guest) and exposed the
client config + a ClientHello driver. net-7c-2 adds the real transport:
`TlsStream<S>` drives the unbuffered rustls state machine over any blocking
`Read + Write` byte stream (a `/net` `TcpStream`), `load_roots_pem` reads the
baked CA bundle, and `loopback_roundtrip` runs a full client<->server handshake +
app-data exchange in memory -- the deterministic, peer-independent in-guest proof
that the real WebPki certificate verification (against the trust anchors + the
wall clock) and the record layer compose end to end. The native `/bin/https` GET
tool is the first consumer.

## Public API

```rust
// Build a TLS 1.3 client config: the RustCrypto provider, the LS-K wall clock,
// and `roots` as trust anchors (the baked CA bundle).
pub fn client_config(roots: rustls::RootCertStore) -> Arc<rustls::ClientConfig>;

// Drive the handshake to its first flight and return the ClientHello record.
// Proves the provider + CSPRNG + ECDHE key share + record layer run on the
// target. Returns None on any state-machine error.
pub fn client_hello(host: &str) -> Option<alloc::vec::Vec<u8>>;

// net-7c-2:

// Parse a PEM CA bundle (the baked /etc/ssl/certs/ca-certificates.crt) into a
// RootCertStore. Fails closed (Config error) on an empty/unparseable bundle.
pub fn load_roots_pem(pem: &[u8]) -> Result<rustls::RootCertStore, TlsError>;

// A client config that trusts NO roots (the negative-test config: every server
// cert fails to verify -- proves the verifier is not a permissive accept-all).
pub fn client_config_untrusting() -> Arc<rustls::ClientConfig>;

// A TLS 1.3 server config from one PEM cert chain + key (the loopback E2E's
// server end; netd serves raw TCP, so a real Thylacine TLS server is later).
pub fn server_config_single_cert(cert_pem: &[u8], key_pem: &[u8])
    -> Result<Arc<rustls::ServerConfig>, TlsError>;

// A TLS client connection over a blocking byte stream S (a /net TcpStream).
// `connect` runs the handshake (validating the server cert vs the config's
// trust anchors + the wall clock); the io::Read/Write impls then carry
// plaintext app data, encrypting/decrypting through the record layer.
pub struct TlsStream<S> { /* ... */ }
impl<S: Read + Write> TlsStream<S> {
    pub fn connect(sock: S, host: &str, cfg: Arc<rustls::ClientConfig>)
        -> Result<TlsStream<S>, TlsError>;
    pub fn close(&mut self);              // best-effort close_notify
}
// impl io::Read + io::Write for TlsStream<S>

// net-8c-2: the symmetric SERVER transport over a blocking byte stream S (a /net
// TcpStream from TcpListener::accept). `accept` runs the server side of the
// handshake (presenting cfg's single cert; no client auth); the io::Read/Write
// impls then carry plaintext app data. Shares ONE private TlsTransport<S, C>
// driver with TlsStream (the client) -- zero record-layer duplication.
pub struct TlsServerStream<S> { /* ... */ }
impl<S: Read + Write> TlsServerStream<S> {
    pub fn accept(sock: S, cfg: Arc<rustls::ServerConfig>)
        -> Result<TlsServerStream<S>, TlsError>;
    pub fn close(&mut self);              // best-effort close_notify
}
// impl io::Read + io::Write for TlsServerStream<S>

// Run a full client<->server handshake + a request/response app-data exchange
// entirely in memory (no sockets). Returns (server_saw, client_saw). Exercises
// the real WebPki verification against client_cfg's anchors + the wall clock.
pub fn loopback_roundtrip(
    client_cfg: Arc<rustls::ClientConfig>, server_cfg: Arc<rustls::ServerConfig>,
    host: &str, request: &[u8], response: &[u8],
) -> Result<(Vec<u8>, Vec<u8>), TlsError>;
```

## Implementation

The crate is `#![no_std]` + `extern crate alloc`. Three pieces make rustls run
on bare metal:

1. **Custom getrandom backend.** `register_custom_getrandom!(thyla_rng)` routes
   `getrandom` to the kernel CSPRNG via `libthyla_rs::t_getrandom` (`SYS_GETRANDOM`).
   The RustCrypto provider draws the client random + the ECDHE ephemeral key
   share through getrandom, which has no `aarch64-unknown-none` backend. The
   registration lives here (the only getrandom user in native userspace), so
   every `tls` consumer inherits it; the macro defines `__getrandom_custom`,
   linked only into binaries that depend on `tls`.

2. **`TimeProvider` over LS-K `CLOCK_REALTIME`.** no_std rustls has no built-in
   clock; webpki reads the wall time to bound a server cert's validity window.
   `ThylaTime::current_time` returns `UnixTime::since_unix_epoch(SystemTime::now().since_epoch())`.
   This is the structural reason TLS follows net-7a (the clock work): a native
   TLS client cannot validate a certificate without a real wall clock.

3. **`builder_with_details(provider, time_provider)`.** The no_std entry point;
   the std `ClientConfig::builder()` (which supplies a default provider + clock)
   is `std`-gated. The connection type is likewise the no_std
   `UnbufferedClientConnection` (the std `ClientConnection` wraps `std::io`).

`client_hello` constructs an `UnbufferedClientConnection`, calls
`process_tls_records(&mut [])` (no incoming bytes yet), and on the resulting
`ConnectionState::EncodeTlsData` calls `encode(&mut out)` -- yielding the
ClientHello TLS record (`out[0] == 0x16`, the handshake content type).

### Cargo recipe (the load-bearing feature flags)

```toml
rustls = { version = "0.23", default-features = false, features = ["custom-provider"] }
rustls-rustcrypto = { version = "0.0.2-alpha", default-features = false, features = ["alloc", "zeroize"] }
rustls-pki-types = { version = "1", default-features = false, features = ["alloc"] }
getrandom = { version = "0.2", features = ["custom"] }
```

- `custom-provider` drops the built-in aws-lc-rs/ring providers (both compile
  C/asm and do not build bare-metal).
- `rustls-rustcrypto` `default-features = false` drops its `std` (which cascades
  `getrandom/std`, breaking the no_std build with `extern crate std`).
- **`tls12` is intentionally omitted** -> TLS 1.3-only (size; see below).

### The unbuffered driver (net-7c-2)

rustls's no_std connection is a pull state machine: feed it the TLS bytes
received so far, it yields what to do next (`EncodeTlsData`, `TransmitTlsData`,
`ReadTraffic`, `BlockedHandshake`, `WriteTraffic`, `PeerClosed`, `Closed`).
`process_tls_records` is defined per role (client/server), but every state it
yields -- and every method on those states -- is generic over the connection's
`Data`, so the post-process logic lives **once** in a shared `drive_state<D>`
helper, and only the `process_tls_records` call differs by role (the
`tlsconn_role!` macro stamps a `Drive` trait impl -- `pump`/`write_app`/
`queue_close` -- for each role's `TlsConn`). The borrow dance: the call returns a
state that borrows both the connection and the incoming buffer, so the body
destructures `self` into disjoint field borrows
(`conn`/`incoming`/`outgoing`/`inbox`), consumes the state into owned results
(staging outgoing TLS bytes + draining decrypted app data), then trims the
incoming buffer -- a `min(discard, len)` clamp guards the trim (a `drain` past
the end would panic, and `panic=abort` turns that into a self-DoS on a path that
talks to hostile servers).

**The blocking transport (net-7c-2 client + net-8c-2 server).** The handshake
loop, the record-layer `Read`/`Write`, and the close path are IDENTICAL for both
roles -- only the constructor (`new_client` vs `new_server`) and the public entry
point (`connect` vs `accept`) differ. So the driver lives **once** in the private
generic `TlsTransport<S, C> where TlsConn<C>: Drive` (the `Drive` bound stays
private because `TlsTransport` is private -- no public signature names it), and
the two public types are thin role-specific wrappers that forward in: `TlsStream`
(client, `connect`) and `TlsServerStream` (server, `accept`). `handshake` loops
pump -> flush outgoing to the socket -> block-read more from the socket, until
the connection settles or the verifier/peer rejects it; it is role-symmetric
(the client's first pump emits the ClientHello to flush; the server's first pump
wants the ClientHello and blocks in `fill_in`). A fix to the transport is a fix
to both roles -- no second copy to drift. `loopback_roundtrip` is the in-memory
alternative: it drives two `TlsConn` drivers cooperatively (no sockets, no
peer), for the deterministic in-process E2E where the blocking transport would
deadlock single-threaded.

**v1.x seam -- no handshake/read deadline (net-7d SA-2 + net-8d F2, #241).**
`handshake` (and `read_plaintext`) loop on `fill_in`, which only errors on a peer
FIN (`n == 0`); there is no overall byte/iteration/wall-clock bound, so a hostile
peer that drips never-advancing TLS bytes loops forever (the `incoming` buffer does
not grow -- the discard-clamp guards it -- but the loop does not terminate). At v1.0
there is NO network-exposed TLS server (`net-echo` is a boot probe with a cooperative
in-guest peer; `https` is a client; `tls-smoke` is in-memory) and the read is
death-interruptible, so this is a forward-seam, not a live defect. Before a real
network-exposed TLS service ships, add a bounded handshake budget + a per-connection
thread/memory bound (slowloris) -- symmetric for both roles in `TlsTransport`.
Related: the fixed 4 MiB userspace heap (net-8d F4, #243) would OOM-abort a
high-volume server -- a growable heap or per-connection memory cap is the paired
Phase-8 service concern.

### The CA bundle + the strip (build wiring)

- **The bundle** is baked into the pool at the canonical
  `/etc/ssl/certs/ca-certificates.crt` (`tools/build.sh::populate_stratum_pool`,
  the same host-bake idiom as `/lib/ndb/local`). The committed source is
  `usr/https/ca-certificates.crt` -- a real Mozilla CA root set (193 roots;
  provenance: the Homebrew `ca-certificates` formula). It is pool data, not a
  compiled-in `webpki-roots` (which would add a second copy of ~150 certs toward
  the binary-size cap). A refresh/update mechanism is a v1.x seam.
- **The strip** (`tools/build.sh::build_ramfs`): the TLS binaries link the full
  rustls + RustCrypto stack and exceed `SYS_SPAWN_BLOB_MAX` (1 MiB)
  *unstripped* (tls-smoke ~1.11 MiB, https ~1.06 MiB), so the kernel could not
  slurp them into a spawn blob. `llvm-strip --strip-all` on the cpio-root COPY
  (-> ~520 KiB each) makes them spawnable; the unstripped artifacts stay under
  `build/usr-rs/.../release/` for manual fault-PC resolution (no userspace tool
  consumes their symbols -- the only addr2line is the kernel's Halls symtab over
  the kernel image). Surgical (TLS bins only); the global workspace keeps
  symbols for every other binary. REVENANT's file-backed exec (task #231)
  retires the cap, after which the strip becomes optional.

## Constraints (net-7c-1 findings)

- **`CAP_CSPRNG_READ` is mandatory.** A `tls` client must be spawned WITH
  `CAP_CSPRNG_READ` (a plain `t_spawn` confers no caps). The spawning authority
  (warden / login / joey) grants it, exactly as joey grants it to corvus.
  Without it `t_getrandom` returns -1, the key share cannot be generated, and
  the handshake fails closed.

- **Binary size at the 1 MiB `SYS_SPAWN_BLOB_MAX` ceiling.** The TLS-1.3 smoke
  is ~1,006,400 bytes (with TLS 1.2 + RSA-kx it is ~1,067,432, OVER the cap; the
  kernel slurps a spawned ELF into a blob bounded by `SYS_SPAWN_BLOB_MAX`).
  v1.0 native TLS is **TLS 1.3-only**. The net-7c-2 size strategy is to strip
  the TLS binaries (no userspace tool consumes their symbols; keep an unstripped
  copy under `build/` for manual fault-PC resolution) -- pure build config, no
  kernel change; a cap bump is the fallback.

- **Set the clock before TLS on RTC-less hardware (net-7d audit F4).** The
  verifier bounds a server cert's validity window against `CLOCK_REALTIME` (via
  the `ThylaTime` provider). On a system whose boot RTC read failed (offset 0 ->
  `CLOCK_REALTIME` == 1970 + uptime), every real CA cert reads as "not yet valid"
  and the handshake **fails closed** -- which is correct (never trust a cert you
  cannot time-bound). The bootstrap path exists because SNTP needs no cert (plain
  UDP NTP): run `sntp <server>` (elevated) to step the clock, *then* TLS works.
  QEMU-virt always has a working PL031, so no in-guest test is affected.

## Status

- **net-7c-1 (landed):** the `tls` crate (provider + getrandom backend + LS-K
  TimeProvider + `client_config` + `client_hello`) + `usr/tls-smoke` (the
  in-guest runtime prover: emits a real ClientHello, asserts it is a valid
  handshake record) + the joey boot probe (spawned with `CAP_CSPRNG_READ`).
  Proven: compile + link + size (1.3-only fits) + runtime (the RustCrypto
  provider, the kernel CSPRNG, and the record layer all execute on aarch64,
  in-guest). Kernel byte-unchanged.
- **net-7c-2 (landed):** the `TlsStream<S>` transport (the unbuffered state
  machine over a blocking `/net` `TcpStream`) + `load_roots_pem` over the baked
  `/etc/ssl/certs/ca-certificates.crt` + `server_config_single_cert` +
  `loopback_roundtrip` + the native `/bin/https` GET tool + the build wiring (the
  CA bundle bake + the TLS-bin strip). The deterministic in-guest proofs: the
  loopback E2E (a full client<->server handshake + cert verify + app-data
  round-trip), the negative test (an untrusted cert is REJECTED -- the verifier
  is not permissive), and the https selftest (the baked bundle parses into 193
  trust anchors + a client config builds). Kernel byte-unchanged.
- **net-7d:** the focused audit of the whole net-7 (incl. the TLS
  cert-validation surface) + the SMP gate + close.
- **net-8c-2 (landed):** the `TlsServerStream<S>` server transport (the symmetric
  counterpart to `TlsStream<S>`, factored over one private `TlsTransport<S, C>`
  driver -- zero record-layer duplication) + `usr/net-echo`'s two-Thread
  TLS-over-`/net` live E2E: a real TLS 1.3 handshake + app-data echo over the
  live `/net` loopback (a server Thread runs the server-side TLS state machine
  while the main Thread drives the client; they interleave through netd, since a
  handshake needs both sides pumping concurrently). The deterministic in-guest
  `TlsStream`-over-socket proof net-7c-2 deferred to net-8. `net-echo` gains
  `CAP_CSPRNG_READ` + joins the TLS-bin strip (it now exceeds
  `SYS_SPAWN_BLOB_MAX` unstripped). Kernel byte-unchanged.

## Tests

- `usr/tls-smoke` (`/bin/tls-smoke`): the in-guest TLS test harness, gated on
  every boot (joey reaps it WITH `CAP_CSPRNG_READ` + asserts exit 0). Three
  proofs: **(1)** `ClientHello OK` (net-7c-1, the crypto path runs); **(2)**
  `loopback E2E PASS` -- a full client<->server handshake + cert verify (against
  a baked self-signed test root + the wall clock) + an app-data round-trip,
  asserting each side saw exactly the other's bytes; **(3)** `untrusted-cert
  REJECTED` -- the negative half: a client with an empty trust store must FAIL
  the handshake, proving the verifier validates the chain (it is not a permissive
  accept-all -- without this, the positive E2E alone could pass with a broken
  verifier). The baked test cert (`usr/tls-smoke/testdata/loopback-{cert,key}.pem`)
  is an ECDSA P-256 self-signed leaf, SAN `DNS:loopback.test`, valid 2020..2044.
- `usr/https` (`/bin/https`, spawned WITH `CAP_CSPRNG_READ`): `https`/`https
  --selftest` is the deterministic gate -- read + parse the baked CA bundle into
  a non-empty trust store + build a client config (no network). `https <host>
  [path]` is the best-effort live GET (resolve via `/net/cs`, handshake over a
  `/net` `TcpStream`, print the status line + byte count); shell-runnable, not a
  boot gate.
- `usr/net-echo` (`/bin/net-echo`, spawned WITH `CAP_CSPRNG_READ`): the net-8c-2
  TLS-over-`/net` live E2E. `tls_over_net_e2e` spawns a server Thread
  (`TlsServerStream::accept` over a `/net` `TcpListener` on the resident lo
  stack) + runs the client (`TlsStream::connect`) on the main Thread, and asserts
  a full handshake (cert verified vs the baked trust anchor + the LS-K wall
  clock) + an app-data echo round-trip **over the live mount** (boot line
  `net-8c-2 TLS-over-/net E2E PASS`) -- the deterministic in-guest
  `TlsStream`-over-socket proof net-7c-2 deferred.

## Performance (NP-2)

The `tlsperf` tool (`usr/tlsperf`; NET-PERF NP-2) measures this crate in-guest:
a full TLS 1.3 handshake over the resident loopback (M4) + a micro-bench of the
exact RustCrypto primitives it uses (M5). Headline (HVF): a handshake is ~2.5 ms
(asymmetric-crypto-bound -- ~2 ECDSA-P256 verifies @ 422 us + the server sign + 2
X25519 agrees @ 57 us); AES-128-GCM runs at ~47 MB/s software (no ARMv8 AES
intrinsics on this target). Every primitive is ~10-16x faster on HVF than TCG ->
TLS is CPU-bound, not IO-bound. Full numbers + the levers (hw AES/SHA, session
resumption) in `docs/NET-PERF.md` section 9.

## Naming rationale

The crate keeps the standard name `tls` -- the reader expects it, and a thematic
rename would obscure intent (the bar from CLAUDE.md "Thematic naming").
