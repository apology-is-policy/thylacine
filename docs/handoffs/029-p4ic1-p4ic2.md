## What landed in this session window

### P4-Ic1: Burrow MMIO-type extension (`72e023f` / `a1c9b6a`)

The foundation for `SYS_MMIO_MAP`. Without a Burrow-of-PA type, the VMA + page-fault dispatch path couldn't install MMIO mappings uniformly with the anon-Burrow flow. P4-Ic1 added the type model + lifecycle integration; P4-Ic2 wired the fault path + syscall.

**Design pivot**: I initially considered three options for MMIO mapping:
- (A) Extend Burrow with `BURROW_TYPE_MMIO` (chosen)
- (B) Bypass Burrow entirely; KObj_MMIO does its own PTE install
- (C) Use Burrow but with kludgy `pages` field for fixed PA

(A) won because:
- The VMA layer + demand-page already understand "VMA → Burrow → backing PA"; extending Burrow types fits the existing model.
- `proc_pgtable_destroy` doesn't free leaf data pages (verified at mmu.c:922 comment "The pgtable walk does NOT free leaf pages"). So MMIO PTEs in the pgtable are structurally safe from accidental `free_pages` — no special-case needed.
- The dual-count Burrow lifecycle (handle_count + mapping_count) maps cleanly to "userspace handle holds the KObj_MMIO; Burrow holds an internal ref so the PA claim survives across handle close + mapping lifetime."

**`struct Burrow` extension**:
```c
enum burrow_type {
    BURROW_TYPE_INVALID = 0,
    BURROW_TYPE_ANON    = 1,
    BURROW_TYPE_MMIO    = 2,    // P4-Ic1
};

struct Burrow {
    u64               magic;
    enum burrow_type  type;
    size_t            size;
    size_t            page_count;
    int               handle_count;
    int               mapping_count;
    struct page      *pages;        // NULL for MMIO
    unsigned          order;
    struct KObj_MMIO *kobj_mmio;    // P4-Ic1 — NULL except for MMIO
    u64               pa;            // P4-Ic1 — 0 except for MMIO
};
```

**`burrow_create_mmio(struct KObj_MMIO *kobj_mmio)`**: takes a `kobj_mmio_ref(km)` so the PA claim survives the caller's eventual `handle_close`. Sets `pages = NULL`, `handle_count = 1` (construction reference, matches anon pattern), `pa = kobj_mmio->pa`. The kobj reference is released in `burrow_free_internal` when both counts reach 0.

**`burrow_free_internal` type-dispatched**:
```c
switch (v->type) {
case BURROW_TYPE_ANON: free_pages(v->pages, v->order); v->pages = NULL; break;
case BURROW_TYPE_MMIO: kobj_mmio_unref(v->kobj_mmio); v->kobj_mmio = NULL; break;
case BURROW_TYPE_INVALID: default: extinction(...);
}
```

`burrow_acquire_mapping`'s liveness check is also type-dispatched (ANON checks `pages != NULL`; MMIO checks `kobj_mmio != NULL`).

**Spec extension to `specs/burrow.tla`**: comment-only. The dual-count `NoUseAfterFree` lifecycle is type-independent (modeled abstractly as VmoIds). The MMIO type's additional cross-spec relationship (holding a ref on KObj_MMIO) is impl-level coordination between `handles.tla::HwResourceExclusive` and `burrow.tla::NoUseAfterFree`. TLC re-verified — correct config explores same state graph; 3 buggy configs produce same expected counterexamples. No regression.

**Tests** (6 new in `kernel/test/test_burrow_mmio.c`): `create_basic`, `create_null_rejected`, `create_holds_kobj_ref` (caller's unref → Burrow ref keeps kobj alive → Burrow unref → kobj freed), `unref_releases_kobj_ref` (reversed ordering), `acquire_mapping_works`, `lifecycle_round_trip` (full create → map → unmap → unref → kobj_unref; verifies destroyed counter + kobj live count return to baseline).

**Posture**: 196 → 202 tests. Default + UBSan green. Reference doc `docs/reference/20-burrow.md` extended with P4-Ic1 MMIO Burrow section.

**Investigation arc — F154 (the P1)**:

The audit agent's prosecution chain was a direct structural mirror of R9 F142:
1. `g_mmio_claims[]` at boot is all-NULL-owners; no kernel MMIO is reserved.
2. P4-Ic5+ grants `CAP_HW_CREATE` to a userspace driver.
3. Driver calls `t_mmio_create(pa=GIC_DIST=0x08000000, size=0x10000, rights=RIGHT_MAP|...)` — succeeds (no overlap with anything in `g_mmio_claims`).
4. Driver calls `t_mmio_map(h, vaddr, prot=READ|WRITE)` — succeeds. Device-memory PTE installed at `vaddr` pointing at `0x08000000` (GIC distributor).
5. Driver writes to GICD_CTLR / GICD_ITARGETSR / GICD_ICENABLER offsets in its user-VA. **Userspace controls kernel IRQ delivery state.**

Fix mechanism: `kobj_mmio_reserve_kernel_ranges()` (in `kernel/mmio_handle.c`) walks DTB for compatibles owned by the kernel (`arm,gic-v3`, `arm,cortex-a15-gic`, `arm,pl011`, `pci-host-ecam-generic`, `virtio,mmio`), page-aligns each range outward, dedupes against prior reservations (so 32 virtio,mmio slots collapse to 4 page reservations), inserts sentinel `KOBJ_MMIO_KERNEL_RESERVED` owner. Subsequent `kobj_mmio_create` overlapping any reserved range returns NULL. Boot banner prints each reservation for observability. Regression test `mmio_handle.kernel_reserved_rejected` locks it.

## What's NEXT — P4-Ic3 onward

**P4-Ic3 — Capability grant path**. The hold-blocker for P4-Ic5+. Currently `rfork` produces children with `caps = CAP_NONE` (via `KP_ZERO`). For a userspace driver to call `t_mmio_create` / `t_mmio_map`, it needs `CAP_HW_CREATE`. At v1.0 only kproc has caps. Three design options:

1. **Kernel-internal `rfork_with_caps(thunk, arg, caps_mask)`**: kproc rforks the driver and explicitly grants `CAP_HW_CREATE`. Simple; matches v1.0 "no userspace cap-grant syscall" stance. Constraint: child's caps_mask must be subset of parent's caps (CapsCeiling per spec). Recommended.

2. **Inherit-on-rfork with mask argument**: `rfork(flags, mask, thunk, arg)` AND's parent caps with mask. More general; deferred to Phase 5+ when the rfork interface gets versioned. Don't do this in v1.0.

3. **Static-grant table**: a kernel-managed list of (binary-path, caps_mask) entries. The exec path consults the table when loading a binary. Plan 9-ish; matches the "/dev/drivers" namespace concept. Deferred to Phase 5+ when stratum is mounted.

Recommendation: **Option 1** for v1.0. ~50 LOC kernel addition; preserves all existing spec invariants; unblocks P4-Ic5 immediately.

**P4-Ic4 — `libthyla-rs` runtime crate extraction**. Extract the inline SVC wrappers from `usr/hello-rs/src/main.rs` into a shared crate `usr/lib/libthyla-rs/` so the virtio-blk driver crate can depend on it. Move `t_exits` / `t_puts` / `t_mmio_create` / `t_mmio_map` / `t_irq_create` / `t_irq_wait` + `_start` (global_asm!) + `#[panic_handler]`. Update `usr/hello-rs/Cargo.toml` to depend on `libthyla-rs`. Add the crate to `usr/Cargo.toml` workspace members.

**P4-Ic5 — `usr/virtio-blk/` Rust no_std driver crate**. The first real userspace driver. Talks to a VirtIO PCI block device discovered by P4-H. ## Bug-hunting / lessons learned

**P4-Ic1 — what we learned**:

1. **The `proc_pgtable_destroy` "doesn't free leaf data pages" guarantee unblocked the design.** Initial concern was that MMIO PTEs (pointing at non-RAM PAs) would trigger `free_pages` on the device PA when the pgtable is torn down. Reading mmu.c:922 confirmed the walker only frees TABLE pages (L0/L1/L2/L3 page-table pages themselves), not the leaf data pages. So MMIO PTEs are structurally safe from accidental free. **Lesson**: verify the cleanup-path's actual semantics before designing around imagined behavior.

2. **Cross-spec relationships don't always need cross-spec modeling.** `burrow.tla` models lifecycle abstractly (VmoIds); `handles.tla` models handle policy. The MMIO Burrow's relationship to KObj_MMIO (refcount sharing) is impl-level coordination — both specs preserve their own invariants independently. **Lesson**: when extending one spec to support another's machinery, ask "does the new behavior break my invariant?" rather than "do I need to model the relationship?"

3. **Type-dispatched lifecycle is cleaner than conditional flags.** The `enum burrow_type` switch in `burrow_free_internal` is more maintainable than a `bool is_mmio` flag because each branch's failure modes (free_pages on NULL vs kobj_mmio_unref on NULL) are different. **Lesson**: prefer structural enums + switches over flag-based conditionals when failure semantics differ.

**P4-Ic2 — what we learned**:

1. **Structural mirroring across audit rounds is a real pattern.** R9 F142 (P0) closed the IRQ side by reserving kernel-INTIDs in `g_intid_claimed[]`. R10 F154 (P1) is the exact structural mirror on the MMIO side — kernel ranges weren't in `g_mmio_claims[]`. The R10 audit agent caught it because the prompt framed P4-Ic2 in terms of "what could a CAP_HW_CREATE caller exploit?" — same framing as R9. **Lesson**: when a chunk introduces a new resource-claim layer, audit whether KERNEL-OWNED instances of that resource are pre-registered. The R9 close gives the template; the R10 audit applied the template.

2. **Dedupe-on-overlap simplifies kernel-range reservation.** Initially the reservation loop hit "g_mmio_claims full" because 32 virtio,mmio slots (each 0x200 bytes) expanded to 32 page-reservations after page-aligning outward. Adding `if (ranges_overlap(...)) return;` to `reserve_kernel_range` collapsed the 32 entries to 4 (one per 0x1000-aligned page). Same correctness; better space efficiency. **Lesson**: when sub-page granularity meets page-aligned tracking, dedupe-on-overlap is the canonical fix.

3. **The "device-memory PTE attrs" question is more architectural than mechanical.** Selecting `MAIR_IDX_DEVICE` (0) vs `MAIR_IDX_NORMAL_WB` (2) is one bit-field write. But the IMPLICATIONS — gather/reorder/coalesce semantics, shareability bits ignored, instruction-fetch implementation-defined per ARM ARM B2.7.2 — drove three R10 findings (F155 W-only AP encoding, F157 EXEC+device, F158 vaddr bit-47). **Lesson**: when introducing a new MAIR index, walk through ARM ARM B2.7 with each operation (R, W, X) + each shareability variant + each AP encoding and document which combinations are valid/meaningful. The defense-in-depth checks at multiple layers (syscall, PTE encoder) catch each path.

4. **AArch64 has no W-only AP encoding.** This was a surprise — POSIX `PROT_WRITE` without `PROT_READ` is a meaningful state on x86 (via PAE / 2-bit AP variant) but not on AArch64 (4-state encoding: {RW EL1, RW any, RO EL1, RO any}). Any v1.0 driver that requested W-only would have gotten an effectively RW PTE — silent rights violation. F155 closed by rejecting at the syscall layer. **Lesson**: when designing capability-bounded mappings, verify the underlying ISA encoding can REPRESENT every (rights, prot) combination the model claims is meaningful.

5. **The "audit closes WITH the chunk" pattern keeps history clean.** Both P4-Ib (R9) and P4-Ic2 (R10) followed the same pattern: substantive impl → run audit → fold fixes back into the SAME substantive commit before merge. This is `--no-amend`-friendly (no rewriting; we just hold the substantive in the working tree until the audit closes) and produces ONE commit per chunk in `git log`. **Lesson**: hold the substantive commit until the audit fixes are folded in. The hash-fixup follows as a separate trivial commit. Two commits per audit-bearing chunk, not three or four.
