**Tip**: `d811c0f` (P4-Ic5b2 hash fixup) on `main`.

This window closes the **last substrate gate** before userspace drivers can run + lands the **first composed userspace driver** that proves the v1.0 hw-handle surface is sufficient for real device work:

1. **P4-Ic5b1b** (`0891c4a` substantive / `7e747fa` hash fixup) — DMA-pinning syscall + `KObj_DMA` + commentary-only spec extension. ## What landed, per chunk

### P4-Ic5b1b (`0891c4a` substantive, `7e747fa` hash fixup)

**Closes**: the last audit-bearing substrate gate before driver crates. Adds the third hw-kobj kind in the family (`KObj_MMIO`, `KObj_IRQ`, now **`KObj_DMA`**).

**Mechanism**: kernel-allocated contiguous pinned page chunks, returned to userspace as a handle whose `pa` field is stable for the handle's lifetime.

1. New `kernel/include/thylacine/dma_handle.h` + `kernel/dma_handle.c` (~180 LOC):
   - `struct KObj_DMA { magic, pa, size, pages, order, ref }`.
   - `kobj_dma_create(size)`: page-aligns `size` up + bounds at `KOBJ_DMA_MAX_SIZE = 1 MiB` (order 8); `alloc_pages(order, KP_ZERO)` for a contiguous chunk; `k->pa = page_to_pa(pages)`.
   - `kobj_dma_ref` / `kobj_dma_unref`: ACQ_REL atomic refcount; magic clobber on free; live counter for diagnostics. Mirrors `kobj_mmio`'s R9 F148 discipline.
2. **No `g_dma_claims` table** — unlike MMIO, the buddy allocator's per-allocation partitioning IS the claim layer (each `alloc_pages` returns a fresh chunk; distinct `KObj_DMA` cannot share PA structurally — stronger than MMIO's explicit overlap-table). PA stability across handle lifetime is a structural impl property (no kernel code mutates `k->pa` post-create; no migrator exists at v1.0). Documented in `specs/SPEC-TO-CODE.md` cross-references.
3. New `BURROW_TYPE_DMA = 3` enum value + `kobj_dma` field on `struct Burrow`; `burrow_create_dma(kobj_dma)` wraps the KObj_DMA with the construction-ref pattern from P4-Ic1. `burrow_free_internal` + `burrow_acquire_mapping` type-switch dispatch to the DMA case → `kobj_dma_unref`.
4. `arch/arm64/fault.c::userland_demand_page` dispatches `BURROW_TYPE_DMA` → `page_pa = burrow->pa + intra-burrow-offset; device_memory = false` (Normal cacheable PTE attrs; CPU + device both coherent on QEMU virt's VirtIO transports — no explicit cache maintenance needed at v1.0).
5. New `SYS_DMA_CREATE = 6` + `SYS_DMA_MAP = 7` syscalls: cap-gated `CAP_HW_CREATE` (HwHandleImpliesCap defense-in-depth); rights validation; W^X enforcement (EXEC rejected); W-without-R rejected (R10 F155 mirror — AArch64 has no W-only AP encoding); `SYS_DMA_MAP` returns PA on success so the driver can embed it into device-visible descriptors.
6. `kernel/handle.c::handle_release_obj` + `handle_acquire_obj` integrate KOBJ_DMA → `kobj_dma_unref` / `kobj_dma_ref`.
7. libthyla-rs (Rust) + libt (C) gain `t_dma_create` + `t_dma_map` SVC wrappers.

**Spec posture**: commentary-only update to `specs/handles.tla` + cross-references in `specs/SPEC-TO-CODE.md`. The existing `HwKObjs` partition + 5 hw invariants (HwHandlesAtOrigin, NoHwDup, HwResourceExclusive, HwHandleImpliesCap, RightsCeiling) apply uniformly to `KObj_DMA` **without new state, actions, or invariants** — the spec already covered DMA via the partition introduced at P4-Ib. PA-stability is a structural impl property (no migrator), not spec-modeled. TLC sanity verified post-commentary (handles_buggy_elevate.cfg counterexample at expected depth 4).

**Tests**: 13 new across `dma_handle` (7) + `burrow_dma` (4) + `dma_map` (2): 213 → 226.

**Audit footprint**: audit-bearing per CLAUDE.md trigger surfaces (Handle table + VMO + Capability checks + Page fault demand-page). Self-audit found no findings across lifetime / lock ordering / error-path cleanups / idempotency / integer overflow / state-machine guards / boundary conditions. **R12-DMA formal pass deferred-or-on-finding.**

**LOC**: ~350 kernel + ~80 tests + ~60 userspace wrappers + ~80 docs/spec = ~570 LOC total.

**Trade-offs accepted**:
- 1 MiB max per allocation (sufficient for VirtIO rings + indirect tables + bounce buffers; Phase 5+ may raise if a single driver needs more).
- Round-up wastes pages on non-power-of-two requests (mirrors `burrow_create_anon`'s approach).
- PA-stability is structural-not-spec-modeled (existing `HwKObjs` partition already pins what's needed; no need for a `kobj_pa` state variable).

### P4-Ic5b2 (`fe40f1b` substantive, `d811c0f` hash fixup)

**Closes**: first composed userspace driver. Closes DMA-side SVC-path coverage in real driver exercise. Proves the v1.0 hw-handle surface (MMIO + IRQ + DMA + caps) is sufficient for real device work.

**Mechanism**: Rust no_std crate using libthyla-rs; drives QEMU virtio-blk through VIRTIO 1.2 modern MMIO transport end-to-end.

The probe pipeline (per VIRTIO 1.2 §3.1.1 + §5.2):

1. **Claim virtio-mmio bank** (4 pages, since slots are 0x200 = 512 bytes; 8 per page; can't claim a single slot in isolation). For each of the 4 pages at PA `0x0a000000 + i*0x1000`: `t_mmio_create(pa, 0x1000, READ|WRITE|MAP)` + `t_mmio_map(handle, va, READ|WRITE)`. Maps to user-VA `0x500000..0x504000`.
2. **Scan slots 0..31 for `DeviceID == 2`** (block). For each slot, read magic at slot offset 0x000 (must == 0x74726976 "virt"), then device_id at offset 0x008. The found slot at v1.0 wiring lands at index 31 (PA 0x0a003e00) because QEMU virt's `hw/arm/virt.c` assigns virtio-mmio slots in REVERSE creation order, and `tools/run-vm.sh` lists `-device virtio-blk-device,drive=disk0` first.
3. **Derive INTID**: `INTID = 48 + slot` (QEMU virt maps virtio-mmio slot k to SPI 16+k → GIC INTID 32 + (16+k)). For slot 31 → INTID 79.
4. **Verify Version == 2** (modern MMIO transport); legacy rejected. Legacy uses a different register layout with `GUEST_PAGE_SIZE` / `QueuePFN` semantics that the probe does not support.
5. **Subscribe to IRQ BEFORE enabling the device**: `t_irq_create(intid, T_RIGHT_SIGNAL)`. This ensures `gic_attach` + `gic_enable_irq` are in place by the time the device fires its first completion IRQ. Inside `kobj_irq_create` (post-P4-Ic5b2): `gic_set_spi_edge_triggered(intid)` flips `GICD_ICFGR` to edge-triggered BEFORE `gic_enable_irq` (see "GIC ICFGR transition" below).
6. **Allocate one 4 KiB DMA buffer**: `t_dma_create(0x1000, READ|WRITE|MAP)` + `t_dma_map(handle, va, READ|WRITE)`. Returns PA for descriptor-embed. Layout inside the buffer (fixed offsets):
   - `+0x000` (256 B): descriptor table (16 entries × 16 B; 16-aligned per spec).
   - `+0x100` (~38 B): avail ring (`le16 flags, le16 idx, le16 ring[16]`).
   - `+0x200` (~134 B): used ring (`le16 flags, le16 idx, used_elem ring[16]`; each used_elem is `le32 id + le32 len`).
   - `+0x300` (16 B): virtio_blk request header (`le32 type, le32 reserved, le64 sector`).
   - `+0x400` (512 B): data buffer (one sector).
   - `+0x600` (1 B): status byte.
7. **VIRTIO 1.2 init sequence** (binding order per §3.1.1):
   - Write Status = 0 (RESET).
   - Write Status |= ACKNOWLEDGE = 1.
   - Write Status |= DRIVER = 2.
   - DeviceFeaturesSel = 0; read DeviceFeatures (low 32 bits; we don't claim any of these). DeviceFeaturesSel = 1; read DeviceFeatures (high 32 bits); verify bit 0 (VIRTIO_F_VERSION_1 = bit 32 overall) is set.
   - DriverFeaturesSel = 0; write DriverFeatures = 0. DriverFeaturesSel = 1; write DriverFeatures = bit 0 (claim VIRTIO_F_VERSION_1).
   - Write Status |= FEATURES_OK = 8; re-read Status; verify FEATURES_OK still set (per §3.1.1 — device may reject our subset).
   - QueueSel = 0; verify QueueNumMax >= 16; write QueueNum = 16.
   - Write QueueDescLow/High + QueueDriverLow/High + QueueDeviceLow/High with the DMA buffer offsets.
   - Write QueueReady = 1.
   - Write Status |= DRIVER_OK = 4. Device may now accept requests.
8. **Submit `VIRTIO_BLK_T_IN(sector=0)`** via 3-descriptor NEXT-linked chain:
   - desc[0]: addr = dma_pa + REQ_OFF, len = 16, flags = NEXT, next = 1 (request header; OUT — device reads).
   - desc[1]: addr = dma_pa + DATA_OFF, len = 512, flags = NEXT | WRITE, next = 2 (data buffer; IN — device writes).
   - desc[2]: addr = dma_pa + STATUS_OFF, len = 1, flags = WRITE (status byte; IN — device writes).
   - Write avail.ring[0] = 0 (head of chain), avail.flags = 0.
   - `dsb sy` (memory barrier; descriptor + req header writes must be visible before idx bump).
   - Write avail.idx = 1.
   - `dsb sy` again before MMIO notify (idx bump must be visible before kick).
   - Write QueueNotify = 0 (queue index). This kicks the device.
9. **Sleep on `t_irq_wait(irq_handle)`**. Returns when the device's completion IRQ fires + kobj_irq_dispatch wakes the Rendez.
10. **ACK the device's interrupt**: read InterruptStatus, write same bits back to InterruptACK. This deasserts the device's IRQ line + clears its pending bit per VIRTIO 1.2 §4.2.2.
11. **Verify completion**:
    - `used.idx == 1` (one entry written by device).
    - `used.ring[0].id == 0` (the descriptor head we submitted).
    - `used.ring[0].len == 513` (informational; 512 data + 1 status).
    - Status byte at `+0x600` == 0 (`VIRTIO_BLK_S_OK`).
    - Data buffer first 16 bytes match `b"THYLACINE-DISK-1"`.

**Build + run wiring**:
- New `tools/build.sh disk` produces `build/disk.img` (1 MiB raw, block 0 = 16-byte ASCII signature + zeroes). Wired into `build_kernel` chain so `tools/test.sh` always has a disk to attach.
- `tools/run-vm.sh` attaches:
  ```
  -drive if=none,id=disk0,format=raw,file=$DISK_IMG
  -device virtio-blk-device,drive=disk0
  -global virtio-mmio.force-legacy=false
  ```
  The `-global` flag is **non-obvious**: QEMU 10.x on the ARM virt machine defaults virtio-mmio to the legacy transport (Version=1) for backward compat with pre-VIRTIO-1.0 kernels. Without the flag, `MagicValue` reads as expected but `Version` reports 1 and the modern-transport-only probe rejects.

**GIC ICFGR transition** (audit-bearing):
- QEMU virt's `hw/arm/virt.c::create_virtio_devices` declares virtio-mmio interrupts as `GIC_FDT_IRQ_FLAGS_EDGE_LO_HI` in the DTB. The Thylacine GIC init (`arch/arm64/gic.c::gic_init`) defaults all SPIs to level-triggered (`ICFGR = 0`); without flipping to edge for virtio-blk's SPI, the device's completion IRQ either fires infinitely (level-after-edge, depending on GIC impl) or fails silently (some GIC impls strictly require trigger-mode match).
- New `gic_set_spi_edge_triggered(intid)` in `arch/arm64/gic.{c,h}` does RMW on `GICD_ICFGR(intid/16)` to set the 2-bit field at position `(intid % 16) * 2` to `0b10` (edge-triggered).
- `kernel/irqfwd.c::kobj_irq_create` calls it for SPIs >= 32 BEFORE `gic_attach` + `gic_enable_irq`. ICFGR writes are only permitted while the INTID is disabled (per ARM IHI 0069 §12.9.7), which we satisfy structurally — the INTID is just claimed via `intid_try_claim` and is not yet enabled.

**Scheduler deadlock workaround** (audit-bearing):
- When the virtio-blk-probe child blocks in `t_irq_wait` waiting for the device's IRQ AND the parent test thread is also blocking (in `wait_pid`), the scheduler's "no runnable peer system-wide" check in `kernel/sched.c §"Block path"` ELE's. The check is correct for software-only wake protocols (where every Rendez has a waker scheduled by another thread) but has no model for "hardware IRQ will wake us."
- Workaround in `kernel/test/test_virtio_blk_probe.c`: replace `wait_pid` with a yield-poll pattern:
  ```c
  for (;;) {
      struct Proc *c = proc_find_by_pid(pid);
      if (!c || c->state == PROC_STATE_ZOMBIE) break;
      sched();
      __asm__ __volatile__("wfi" ::: "memory");
  }
  int reaped = wait_pid(&status);  // sees zombie immediately; doesn't block
  ```
  The parent stays RUNNABLE (running its yield loop), so the scheduler always has a peer to pick. WFI in the loop lets the CPU idle between yields, allowing QEMU's virtio-blk simulation to progress on the host thread and fire the IRQ in normal time (~µs to ms). Without the WFI, the parent busy-spinning at 100% CPU starved QEMU's virtio thread and pushed boot-time to ~2 s instead of ~440 ms.
- **Long-term fix (R12-sched deferred audit)**: replace the ELE with WFI-and-retry in the scheduler's deadlock branch. Timer IRQ at 1 kHz bounds the retry latency; the change touches a load-bearing scheduler invariant and should be spec-first (extend `scheduler.tla` with a `HardwareWake(t)` action modelling external IRQs).

**Tests**: 1 new (`userspace.virtio_blk_probe_rfork_with_caps`): 226 → 227. The kernel-side test gracefully skips (returns PASS) if either the probe binary is absent from the ramfs OR no virtio-mmio slot has `DeviceID == 2` (i.e., no `-device virtio-blk-device` in the QEMU invocation). Both gates short-circuit before any rfork, so a developer running without the disk image attached gets a clean SKIP rather than a hang.

**Audit footprint**:
- **Non-audit-bearing at the per-syscall layer**: all four hw-handle syscalls (MMIO_CREATE, MMIO_MAP, IRQ_CREATE, IRQ_WAIT, DMA_CREATE, DMA_MAP) were audited in their landing chunks (R9 for MMIO/IRQ create, R10 for MMIO_MAP, R11 for cap-grant primitive, P4-Ic5b1b self-audit for DMA). The probe just composes them.
- **Audit-bearing at the GIC ICFGR transition** (Scheduler trigger surface + IRQ delivery semantics). Self-audit found no findings across the standard six categories. Lifted to **R12-gic-edge deferred audit**.
- **Audit-bearing at the scheduler-deadlock workaround** (Scheduler trigger surface). The workaround is contained in the test harness; the underlying scheduler limitation is unchanged. Self-audit + documentation. Lifted to **R12-sched deferred audit** (which is the bigger of the two — proper fix requires spec extension).

**LOC**: ~430 userspace + ~145 kernel test + ~40 kernel gic + ~12 kernel irqfwd + ~25 tools/build.sh + ~38 tools/run-vm.sh + ~80 docs ≈ ~770 LOC total.

**Trade-offs accepted**:
- All userspace SPIs unconditionally edge-triggered (no per-IRQ flag). QEMU virt's virtio-mmio + most real-ARM device IRQs are edge per DTB; a future level-triggered consumer (PCIe legacy INTx) would need a flag on `t_irq_create`.
- `gic_set_spi_edge_triggered` RMW is not SMP-safe for concurrent claims of different SPIs in the same 16-INTID window (one register covers 16 INTIDs × 2 bits each). At v1.0 IRQ claims happen serially from a single thread (kproc + test runner); Phase 5+ may need a lock or LL/SC.
- Yield-poll pattern in the test instead of fixing the scheduler. The fix touches load-bearing invariants (LatencyBound, NoMissedWakeup); proper resolution is a separate spec-first chunk.

## Cross-cutting observations

### The substrate is now complete + proven

With KObj_MMIO + KObj_IRQ + KObj_DMA + CAP_HW_CREATE all in place, plus a real driver that composes all of them successfully, **Phase 4's "userspace drivers" exit criterion is structurally satisfied for the read path**. The remaining work in §6.2 is throughput (1 GiB read/write loop) + write-path verification + the other VirtIO devices (net, input, gpu).

### "Why didn't the IRQ fire?" was the longest debug

The most time-consuming debug in P4-Ic5b2 was tracking down why the virtio-blk completion IRQ didn't fire on the kernel side. The symptom was a clean scheduler ELE with "no runnable peer system-wide" after `SYS_IRQ_WAIT entering...`. Three separate issues stacked:

1. **QEMU virtio-mmio defaults to legacy** (Version=1). Without `-global virtio-mmio.force-legacy=false`, the device exposes a different register layout and our modern-only probe rejected with "slot reports legacy version (1); modern (2) required". *Trap-door clue*: the probe got past the magic check + DeviceID match before failing on version.
2. **GIC ICFGR was level-triggered for SPI 47** (the virtio-blk slot's IRQ). QEMU's emulated edge-rising line never produced a level the GIC could sample, so the IRQ never delivered to CPU 0 despite being enabled. *Trap-door clue*: `virtio_mmio_setting_irq virtio_mmio setting IRQ 1` appeared in QEMU's trace output, proving the device-side fired, but kernel-side `kobj_irq_dispatch` was never invoked.
3. **Scheduler deadlock detection too aggressive**. Even after the IRQ delivery was fixed, the timing race (device completes faster than parent reaches `wait_pid`'s sleep vs slower) caused intermittent ELE. The yield-poll workaround keeps the parent RUNNABLE so the scheduler always has a peer.

All three issues are documented inline + as deferred audit notes. The combination is worth highlighting: QEMU defaults, ARM GIC config, kernel scheduler — each platform layer contributed one obstacle.

### Image-size discipline

Adding a 256 KiB static buffer to `test_virtio_blk_probe.c` (mirroring mmio-probe's + irq-probe's sizes) pushed the kernel image_size header past where QEMU's `-initrd` loader places the ramfs. Symptom: QEMU silently produced zero serial output (kernel hung early in `_bss_clear` because it overwrote the initrd). Reduced to 128 KiB. **Lesson**: kernel BSS growth interacts with QEMU's `-initrd` placement. The kernel image header's `image_size` field tells the bootloader how much RAM to reserve; the loader places initrd just above that. Growing the kernel past the placement boundary causes overlap.

A real fix would be to either (a) load `initrd` at a fixed offset well above the kernel (some boards do `-fdt-load-base`), or (b) parse the DTB's `linux,initrd-start/end` properties and tolerate any kernel-end-to-initrd-start window. At v1.0 the simpler fix is to keep BSS bounded; the 128 KiB cap per probe buffer is documented inline.

## Lessons learned

### P4-Ic5b1b

1. **The buddy allocator is the natural claim layer for kernel-allocated resources.** P4-Ic5b1b initially considered an explicit `g_dma_claims[]` table mirroring `g_mmio_claims[]`. But the MMIO claim table exists because userspace specifies the PA range to claim, and overlap detection is the structural invariant. For DMA, the kernel picks the PA via `alloc_pages`, and the buddy allocator's per-allocation partitioning already guarantees distinct chunks never share PA. The explicit table would have been pure redundancy.
2. **Commentary-only spec extensions are valid when the existing spec already pins the new behavior.** The handles.tla `HwKObjs` partition + 5 hw invariants (introduced at P4-Ib for MMIO + IRQ) apply uniformly to a third hw-kobj kind without any state, action, or invariant changes. PA-stability for DMA is a structural impl property (no migrator), not a spec invariant. Adding spec state to model what the impl structurally guarantees would have been over-modeling.
3. **PA stability is a structural-vs-spec distinction worth documenting.** If a future Phase 5+ chunk adds a memory migrator (e.g., for huge-page promotion), DMA PA-stability becomes a spec invariant the migrator must respect. SPEC-TO-CODE.md flags this so the future author knows when to upgrade the spec.

### P4-Ic5b2

1. **VIRTIO 1.2 modern transport is the default to target.** Legacy (Version=1) uses a different MMIO register layout with `GUEST_PAGE_SIZE` and `QueuePFN` semantics that don't fit our 64-bit-PA + arbitrary-page-size model. Real hardware ARM systems uniformly provide modern transport per VIRTIO 1.2; the QEMU legacy default is a backward-compat artifact. Document `-global virtio-mmio.force-legacy=false` inline in `tools/run-vm.sh`.
2. **GIC ICFGR transition is per-INTID, not per-device.** Each device's DTB entry declares its trigger mode independently. The kernel can't blindly default to edge or level; either it consults the DTB at IRQ-create time (Phase 5+ DTB-driven discovery API) or it asks the driver. v1.0 assumes edge for all userspace-claimed SPIs since QEMU virt's virtio-mmio + most real-ARM devices are edge per DTB; documented as the v1.0 default + future-flag-out.
3. **Scheduler deadlock checks need a model for external wake.** The current check is a debugging aid that catches the common bug of "you forgot to register a waker"; it's wrong-by-construction for hardware-driven async waiters. The fix is architecturally clean (WFI-and-retry, like every other OS) but touches a load-bearing scheduler invariant. Worth doing spec-first.
4. **Yield-poll with WFI is the cheap, contained workaround.** No scheduler change needed; the test harness handles the race entirely on its own side. Pattern is generalizable to any future kernel test that needs to wait on a hardware-driven Rendez while another thread is also blocked.
5. **Composition exposes integration bugs that per-syscall tests miss.** The four hw-handle syscalls each had passing per-syscall tests. Composing all four in one process surfaced (a) the GIC ICFGR / DTB mismatch, (b) the QEMU legacy default, (c) the scheduler deadlock race. None of these were detectable from per-syscall tests in isolation. **Lesson**: every Phase 4+ chunk that adds a new hw-handle composition pattern should treat it as audit-bearing at the composition layer even if the individual syscalls are non-audit-bearing.
