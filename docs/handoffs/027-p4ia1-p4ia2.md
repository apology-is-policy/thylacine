# Handoff 027 — P4-Ia1 + P4-Ia2 (C + Rust native userspace runtimes live)

**Tip**: `051c20c` (P4-Ia2 hash fixup). Predecessor: `5e82d85` (handoff 026).

## TL;DR

Two sub-chunks of P4-I landed in this session window:

1. **P4-Ia1** — C userspace runtime + first binary built out-of-tree. Replaces "userspace = a hand-encoded blob embedded in kernel/joey.c" with "userspace = real ELFs built via clang+ld.lld and loaded from cpio at boot". New top-level `usr/` tree as a **sibling CMake project** to the kernel. `libt` (Thylacine userspace runtime) provides `_start` + stack-canary stubs + header-only SVC wrappers. `/hello` is the first binary. `tools/build.sh userspace` implemented (was stub); `build_kernel` chains it before `build_ramfs`. New `devramfs_lookup(name, **data, *size)` helper and a kernel test `userspace.ramfs_hello` exercising the entire pipeline.

2. **P4-Ia2** — Rust nostd userspace + first Rust binary. Cargo workspace under `usr/`; built-in Rust target `aarch64-unknown-none`; `/hello-rs` uses inline `core::arch::asm!` SVC wrappers + `global_asm!`-defined `_start` + `#[panic_handler]`. Shares the same W^X linker script as the C side — both binaries produce ELFs the kernel ELF loader handles identically. Build wrapper extended; second kernel test `userspace.ramfs_hello_rs` validates the Rust path end-to-end.

**Phase 4 userspace substrate live**: native Thylacine binaries can now be authored in both C and Rust, built reproducibly, shipped via the existing P4-E ramfs path, loaded by the existing P3-Eb exec_setup path, and reach `Thylacine boot OK` cleanly. P4-Ib (KObj_MMIO + KObj_IRQ handle-table integration, **spec-first**, audit-bearing) is now next. **Test count: 178 → 180**.

Both chunks are **non-audit-bearing** per CLAUDE.md §"Audit-triggering changes" — toolchain scaffolding doesn't touch a §28 invariant. No new TLA+ specs (the W^X invariant I-12 enforcement remains pinned by the kernel ELF loader's rejection of RWX segments + the static linker-script discipline).

## What landed in this session window

### P4-Ia1: C userspace runtime + first hello binary (`5467725` / `e58cbc3`)

**The problem at chunk start**: `kernel/joey.c` was a 235-line file containing a hand-encoded 9-instruction AArch64 blob, a synthetic ELF wrapper around it, and the rfork + exec_setup + userland_enter + wait_pid machinery. Its own comment said: *"P3-F's successor (Phase 4 ramfs / Phase 5 toolchain) will replace this with a real binary build pipeline."* `tools/build.sh sysroot` and `tools/build.sh userspace` were stubs that errored out with stale phase tags ("Phase 5+" / "Phase 3+"). The ramfs (built by P4-E) shipped only two placeholder text files (`welcome`, `version`).

**What we built**:

`usr/` — top-level **sibling CMake project** to the kernel. Sibling rather than subordinate because cross-compilation toolchain files don't compose cleanly across nested subprojects in one CMake invocation, and because the userspace toolchain has substantially different flags from the kernel (FP allowed, standard code model, no `-mgeneral-regs-only` / `-mcmodel=tiny` / `-fpie`).

```
usr/
├── CMakeLists.txt                 # project(thylacine_userspace)
├── scripts/aarch64-userspace.ld   # W^X linker script (two PT_LOAD)
├── lib/libt/
│   ├── CMakeLists.txt
│   ├── include/thyla/syscall.h    # SVC wrappers (header-only)
│   └── src/start.S                # _start + stack-canary stubs
└── hello/
    ├── CMakeLists.txt
    └── hello.c
```

**`libt` (Thylacine userspace runtime)**:
- `<thyla/syscall.h>` — header-only SVC wrappers: `t_exits(status)` (noreturn, `T_SYS_EXITS=0`), `t_puts(buf, len)` (`T_SYS_PUTS=1`, returns x0), `t_putstr(s)` (convenience wrapper). Inline-asm clobber list is just `"memory" + "cc"` because the kernel saves+restores the full exception_context across SVC.
- `start.S` — `_start`: `bti c` → `bl main` → `mov x8, #0; svc #0` → defensive `wfe; b 1b`. Stack-canary stubs: `__stack_chk_guard` (sentinel `0xdeadbeefcafef00d`) + `__stack_chk_fail` (→ `T_SYS_EXITS(1)`). `.note.GNU-stack` marker for NX stack.

**`aarch64-userspace.ld`** — two PT_LOAD segments (text+rodata RX=5, data+bss RW=6) at base `0x400000` (matches Linux static-executable convention). W^X enforced two ways: this script (no segment carries both W+X) + the kernel ELF loader (rejects RWX). Discards `.eh_frame`, `.dynamic`, `.gnu.hash`, etc. that would bloat without contributing at runtime.

**`cmake/Toolchain-aarch64-userspace.cmake`** — sibling of `Toolchain-aarch64-thylacine.cmake`. Same `clang` + `ld.lld` + triple `aarch64-none-elf`. Diff is conditional flags only (no kernel-internal hardening like `-mgeneral-regs-only`; userspace gets FP/SIMD).

**`tools/build.sh`**:
- `build_userspace`: invokes CMake on `usr/` → `build/usr/`.
- `build_kernel`: now chains `build_userspace` + `build_ramfs` (a single `tools/build.sh kernel` produces a fully populated initrd).
- `build_ramfs`: curates `usr_bins=( hello )` array; each binary copied from `build/usr/<name>/<name>` into `build/ramfs-src/` before mkcpio. Curated rather than glob to prevent shipping CMake byproducts.
- `build_sysroot`: clarified to be the **Linux-compat musl tree** (Phase 6+), distinct from native Thylacine `userspace`. Stub error message now points users at the right command.

**`kernel/include/thylacine/devramfs.h`** (new public header) — exposes `devramfs_lookup(name, **out_data, *out_size)`. Returns pointer aliasing the kernel-owned initrd blob (no caller-side free). Used by the new test and reserved for the future Phase 5+ exec syscall implementation.

**`kernel/test/test_userspace_ramfs.c`** (new) — registers `userspace.ramfs_hello`. Sequence:
1. `devramfs_lookup("hello", &cpio_blob, &size)`. If not found, prints `[skip]` notice and returns PASS (kernel built without userspace tolerated).
2. memcpy from cpio into 8-byte-aligned static buffer (cpio newc aligns to 4; ELF Ehdr cast needs 8 per R5-G F61 / UBSan `-fsanitize=alignment`).
3. `rfork(RFPROC, ramfs_exec_thunk, &args)` → thunk does `exec_setup` + `userland_enter`.
4. `wait_pid` verifies clean exit; TEST_EXPECT_EQ checks pid + status=0.

Boot output (P4-Ia1 tip):
```
[test] userspace.ramfs_hello ...     /hello cpio=0xffff000048000074 → aligned=0xffffa00338cf96a0 size=73776 bytes
    /hello: exec_setup ok entry=0x400000 sp=0x80000000 → userland_enter
hello from /hello (built via tools/build.sh userspace)
    /hello reaped pid=1307 status=0 (build path verified)
PASS
```

**Why not musl**: Musl is a *Linux* libc. Using it for native Thylacine binaries would require porting musl's syscall layer to Thylacine syscall numbers (Plan 9 / 9P / namespace-aware semantics, not Linux). That's a much bigger project than warranted here. Native Thylacine programs talk directly to the Thylacine syscall ABI via libt; musl appears in Phase 6 as the Linux binary-compat tier (per ROADMAP + the user manual's `musl-static` / `musl-dynamic` / `glibc-dynamic` compat hierarchy).

**Why `kernel/joey.c` is retained**: Migrating /joey from embedded blob to disk-loaded is a separate refactor with its own concerns (boot-path liveness during stratum mount transition, the F136 single-call guard's BSS-race role). Doing it in P4-Ia1 would have conflated two concerns. The migration is queued for post-stratum-mount (Phase 5+).

### P4-Ia2: Rust nostd userspace + first Rust hello binary (`48692c6` / `051c20c`)

**The problem at chunk start**: P4-Ia1 established the C path through the build-from-disk pipeline. P4-Ic (the actual virtio-blk driver crate) is Rust — needs the Rust path established. Doing it in one chunk (Ia + Ic combined) would have been too large.

**What we built**:

```
usr/
├── Cargo.toml                     # workspace root
├── .cargo/config.toml             # target + linker pinning
├── Cargo.lock                     # (cargo-generated; committed)
└── hello-rs/
    ├── Cargo.toml
    └── src/main.rs
```

**`usr/Cargo.toml`** — workspace root. Members: `hello-rs`. Release profile: `opt-level = "z"` (size-first; embedded posture), `lto = "fat"`, `panic = "abort"` (no_std has no unwinder), `codegen-units = 1`, `overflow-checks = true` (cheap; surfaces integer bugs in future driver crates). Workspace-package defaults so member crates inherit edition/license/publish.

**`usr/.cargo/config.toml`** — toolchain pinning. Target = `aarch64-unknown-none` (built-in Rust target for bare-metal aarch64; no_std required, no libc, no host OS assumptions). `target-dir = "../build/usr-rs"` (keeps `usr/` git-clean; sibling of `build/usr/` for the C side).

rustflags wire the **shared W^X linker script** (`-Tscripts/aarch64-userspace.ld`) — same script as the C side, so both Rust and C binaries produce ELFs with identical PT_LOAD layout. Posture flags match the C side: `--no-dynamic-linker`, `--build-id=none`, `-z noexecstack`, `relocation-model=static`, `code-model=small`, `target-feature=+lse,+paca,+pacg,+bti`. Cargo runs from `usr/` (where the linker script path resolves correctly).

**`usr/hello-rs/src/main.rs`** — `#![no_std]` + `#![no_main]`. Inline SVC wrappers via `core::arch::asm!`:
- `t_exits(status: i64) -> !` (noreturn, `T_SYS_EXITS = 0`)
- `t_puts(buf: *const u8, len: usize) -> i64` (`T_SYS_PUTS = 1`, returns x0)
- `t_putstr(s: &str) -> i64` (convenience over `&str`)

`_start` is defined via `global_asm!` mirroring libt's C-side `_start`:
```rust
global_asm!(
    "_start:",
    "    bti     c",
    "    bl      rs_main",       // x0 := rs_main()
    "    mov     x8, #0",        // T_SYS_EXITS
    "    svc     #0",
    "1:  wfe; b 1b",
);
```

`rs_main` is `#[no_mangle] pub extern "C" fn rs_main() -> i64` so the AArch64 PCS applies (x0 = return). `#[panic_handler]` calls `t_exits(1)` — the only reasonable response in a no_std program with `panic = "abort"`.

**`tools/build.sh::build_userspace`** — extended to run both:
1. CMake on `usr/` → `build/usr/` (C side).
2. `cargo build --release` from `usr/` → `build/usr-rs/aarch64-unknown-none/release/` (Rust side).

Rust is **skipped gracefully** (with a notice) if `rustup target add aarch64-unknown-none` hasn't been run — the C-only build path stays usable on fresh checkouts. `build_ramfs` curates a second binary list (`usr_rs_bins=( hello-rs )`) copied from `build/usr-rs/<target>/release/`.

**`kernel/test/test_userspace_ramfs.c`** refactored:
- Extracted shared `run_ramfs_binary(name)` helper between `userspace.ramfs_hello` (C) and `userspace.ramfs_hello_rs` (Rust). Same kernel-side machinery (cpio → 8-aligned buffer → rfork → exec_setup → userland_enter → wait_pid); different binary blob.
- Thunk's diagnostic message generalized from hardcoded `"/hello:"` prefix to generic `"exec_setup ok ..."`.

Boot output (P4-Ia2 tip):
```
[test] userspace.ramfs_hello_rs ...     /hello-rs cpio=0xffff00004801211c → aligned=0xffffa002c68f96a0 size=66336 bytes
    exec_setup ok entry=0x0000000000400000 sp=0x0000000080000000 → userland_enter
hello from /hello-rs (Rust no_std, built via cargo)
    /hello-rs reaped pid=1308 status=0 (build path verified)
PASS
```

**Why `aarch64-unknown-none` rather than a custom target spec**: The built-in target gives us a tested, std-installable triple with no_std baked in. Custom target specs are appropriate when the syscall ABI itself diverges from a known target (Phase 5+ when the full Thylacine syscall surface lands and we'd want compiler-level support for the ABI's calling conventions). At v1.0 P4-Ia2 the syscall surface is two SVCs; inline `asm!` is the right tool.

**Why inline SVC wrappers in hello-rs rather than a separate runtime crate**: Only one Rust binary at P4-Ia2. Extracting `libthyla-rs` before the second binary needs the same wrappers would be premature factoring. The extraction is queued for P4-Ic (the virtio-blk driver crate) when the second user emerges.

**Why same linker script as the C side**: The W^X layout requirements are identical — two PT_LOAD segments, text+rodata RX, data+bss RW, base 0x400000. Sharing the script guarantees both paths produce ELFs the kernel loader treats identically. Rust's `hello-rs` happens to emit only ONE PT_LOAD segment (it has no `.data` / `.bss` — no global mutable state in main.rs) which is fine: 1 RX segment passes W^X trivially.

**Why `rs_main` rather than `main`**: Rust's `#![no_main]` disables the `fn main()` lang item; reusing the name would shadow a special-cased identifier. The explicit rename makes the entry point's nature visible at the symbol level and avoids confusion with the C `main` in `usr/hello/hello.c`.

**Posture (P4-Ia2 tip)**: 180/180 PASS × default (~411 ms) + UBSan (~440 ms). 4/4 fault. 5/5 KASLR under `BOOT_TIMEOUT=20`.

## Bug-hunting / lessons learned

**P4-Ia1 — what we learned**:

1. **The cpio newc / ELF alignment mismatch is real, surfacing through R5-G F61.** cpio newc mandates 4-byte data alignment; the kernel ELF loader's `Elf64_Ehdr` cast requires 8 (per UBSan `-fsanitize=alignment`). First test run failed with `exec_setup rc=-1` (= `ELF_LOAD_BAD_ALIGN`). The fix at the test level — memcpy into an 8-byte-aligned static buffer — is the same pattern every future binary-loading caller will need (Phase 5+ exec-from-disk, network loaders, etc.). Documented as caveat #1 in `docs/reference/38-userspace.md` so the lesson doesn't get re-learned the hard way.

2. **Curated binary lists in `build_ramfs` beat glob.** A glob would have shipped CMake byproducts (`CMakeFiles/`, `cmake_install.cmake`, etc.) into the cpio. Explicit `usr_bins=( hello )` arrays make it obvious what's expected to ship; adding a new binary is one array entry + one CMakeLists.txt.

3. **Sibling CMake projects keep toolchain flag sets clean.** Trying to nest the userspace tree as a subdirectory under the kernel's CMakeLists.txt would have meant per-target flag overrides for every userspace TU (because `add_compile_options(${THYLACINE_KERNEL_C_FLAGS})` at top-level applies to all subdirs). Two invocations are cleaner.

4. **"sysroot" naming had to be reclaimed**. The pre-Ia1 `tools/build.sh sysroot` stub claimed "Phase 5 deliverable" but blurred the distinction between (a) the native Thylacine userspace tree (now `usr/`) and (b) the Linux-compat musl sysroot (Phase 6+). Sub-chunk decision: keep `sysroot` reserved for the musl tier; native userspace gets the `userspace` target. The error message now points users at the right command.

**P4-Ia2 — what we learned**:

1. **Rust's built-in `aarch64-unknown-none` target was exactly the right level of abstraction**. Custom target specs would have been overkill for a hello-world; aarch64-unknown-linux-* would have linked too much (libc, even with `+crt-static`); aarch64-unknown-none gives no_std + no host assumptions + a pre-installable rustup target. The target was downloaded and the binary linked in under 3 minutes from a fresh state.

2. **Linker-script reuse between C and Rust just works**. rust-lld treats `-T<script>` the same way ld.lld does. The C-side `usr/scripts/aarch64-userspace.ld` produces valid ELF layouts for the Rust binary too — no Rust-specific tweaks needed. This is a nice property; future Rust binaries don't fragment the layout invariants.

3. **`#![no_main]` requires defining `_start` somewhere**. The C-side approach (a libt.a containing `start.S`) doesn't work for Rust because no_main means cargo doesn't link against an external `_start` provider. Solution: `global_asm!` in `main.rs` defines `_start` directly. The asm calls into Rust via `bl rs_main` where `rs_main` is `#[no_mangle] pub extern "C" fn`. Naming `main` rather than `rs_main` is tempting but shadows the special-cased `main` identifier Rust disabled.

4. **`cargo build` invocation CWD matters**. The `usr/.cargo/config.toml` references the linker script as `-Tscripts/aarch64-userspace.ld` (relative path). rust-lld resolves it relative to cargo's CWD, which is the workspace root. `tools/build.sh::build_userspace` does `( cd "$REPO_ROOT/usr" && cargo build --release )` to ensure this stays correct — without the `cd`, the relative path would resolve to wherever the script was invoked from.

5. **Rust's no_std hello binary is SMALLER than the C one** (66 KB vs 74 KB). Two reasons: rust-lld strips DWARF more aggressively, and the C side has stack-canary stubs (~50 bytes of overhead). Both binaries' actual runtime footprint is tiny (<200 bytes); the rest is unstripped debug.
