## What landed

### P3-C: page-fault dispatcher (`12ff454`)

New `arch/arm64/fault.{h,c}` (~280 LOC) with structured fault classification + dispatcher. Refactored `arch/arm64/exception.c::exception_sync_curr_el` to use the dispatcher.

- `struct fault_info`: decoded view of ESR_EL1 / FAR_EL1 / ELR_EL1 (vaddr, elr, raw esr, ec, fsc, fault_level + booleans from_user / is_instruction / is_write / is_translation / is_permission / is_access_flag).
- `enum fault_result`: FAULT_HANDLED / FAULT_FATAL / FAULT_UNHANDLED_USER.
- `fault_info_decode`: pure bit-extraction.
- `arch_fault_handle`: priority-ordered classification (kstack guard → W^X → translation → permission → access-flag → user → catch-all).

`exception.c` is now a thin dispatcher for data/instruction aborts. The 4 fault-matrix integration tests (kstack_overflow / wxe_violation / bti_fault / canary_smash) all still PASS through the new dispatcher.

5 new unit tests in `kernel/test/test_fault_decode.c` exercise the decoder for representative ESR encodings (kernel-data-translation-L2, kernel-data-permission-write, user-data-translation, user-instruction-fetch, access-flag-fault).

### P3-Da: VMA tree data structure + per-Proc integration (`a7ff570`)

Per-Proc VMA list anchored at `struct Proc.vmas`. A VMA describes a contiguous user-VA range `[vaddr_start, vaddr_end)` with permissions and a backing VMO. Sorted doubly-linked list at v1.0 (O(N) ops); RB-tree is a Phase 5+ optimization.

- `struct Vma`: 64 bytes (magic at offset 0; vaddr_start/end; prot; vmo + offset; next/prev list pointers).
- `vma_init / vma_alloc / vma_free / vma_insert / vma_remove / vma_lookup / vma_drain` API (kernel/vma.c).
- struct Proc grows 112 → 120 bytes (+ struct Vma *vmas). Assert updated.
- Bootstrap: vmo_init → **vma_init** → asid_init → proc_init.
- proc_free calls vma_drain BEFORE handle_table_free for proper VMO lifecycle:
  - vma_drain → vmo_unmap chain (mapping_count--).
  - handle_table_free → vmo_unref chain (handle_count--).
  - vmo_free runs only when both reach 0 (specs/vmo.tla NoUseAfterFree).
- Constraint validation in vma_alloc: page-aligned, ordered, NULL-vmo-reject, W^X-reject (mirrors ARCH §28 I-12 at a third layer).

6 new unit tests in `kernel/test/test_vma.c` cover alloc/free, constraints, insert+lookup, overlap rejection, sorted invariant, and drain releases all.

**DEFERRED to P3-Db**: vmo_map signature extension to take `(Proc, vaddr, length, prot)` → calls vma_alloc + vma_insert.

**DEFERRED to P3-Dc**: arch_fault_handle's user-mode dispatch path → vma_lookup → demand paging.

**DEFERRED to P3-D (proc_pgtable_destroy extension)**: walk + free L1/L2/L3 sub-tables. Currently only frees L0 (no sub-tables exist until vmo_map populates user-half PTEs).

## Trip-hazards added at P3-C + P3-Da (cumulative #125-130)

### P3-C

125. **`exception_sync_curr_el` only sees Current EL/SPx aborts** — EC_DATA_ABORT_LOWER (0x24) and EC_INST_ABORT_LOWER (0x20) don't reach this handler. P3-E adds `exception_sync_lower_el` for the EL0 sync vector; that handler reuses `arch_fault_handle` directly.

126. **`arch_fault_handle` priority order is load-bearing** — kstack guard (#1) is checked BEFORE W^X (#2) so a guard-page access via direct-map alias is recognized correctly even if the access classifies as a permission fault. The order embodies the "specific-then-general" pattern.

127. **Diagnostic strings are stable + grepped by test-fault.sh** — `"kernel stack overflow"`, `"PTE violates W^X (kernel image)"`, `"BTI fault (...)"`, `"stack canary mismatch"` are part of the test-tooling ABI. Renaming requires coordinated update of `tools/test-fault.sh`, `TOOLING.md`, and `CLAUDE.md`.

### P3-Da

128. **struct Proc grows to 120 bytes** — `_Static_assert(sizeof(struct Proc) == 120)` pinned. P3-Db doesn't grow further (just wires existing vmas pointer).

129. **vma_alloc rejects W+X** — third layer of ARCH §28 I-12 enforcement (alongside PTE constructors + ELF loader). Future P3-Db / P3-Dc / P3-E paths must NOT bypass vma_alloc to avoid the check.

130. **proc_free calls vma_drain BEFORE handle_table_free** — order matters for VMO lifecycle (mapping_count must be released before any vmo_unref might cross zero). Reordering would break specs/vmo.tla NoUseAfterFree.
