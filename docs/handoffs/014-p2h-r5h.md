## Trip-hazards added since handoff 013

### P2-H closing audit + spec finalization

- **`current[stealer] = NULL` precondition on Steal** (P2-H scheduler.tla refinement). This closes a spurious lasso at the spec level. The impl's try_steal is only called from pick_next inside sched() — when the calling CPU is between releasing prev and picking next. Future impl changes that introduce a try_steal callsite where the calling CPU has a settled `current` would diverge from this precondition; would need the spec refined to support the new pattern.
- **Per-thread fairness (Yield(cpu, t)) deferred to Phase 5+**. The minimal-universe (2T × 1C) scheduler_liveness.cfg proves LatencyBound under the simpler `Yield(cpu)` parameterization. At full universe (3T × 2C) the CHOOSE-determinism in Yield/Block produces spec-level lassos that don't reflect impl behavior (the impl's EEVDF deadline-driven pick rotates through threads naturally). Full-universe per-thread fairness requires `Yield(cpu, t)` parameterization + `\A cpu, t : SF_vars(Yield(cpu, t))` — meaningful when sched_setweight introduces weight asymmetry (Phase 5+).
- **F75's SYNCHRONIZATION INVARIANT** documented in `kernel/proc.c::proc_link_child` comment block. The invariant: at v1.0, child-list mutators don't take a lock because (a) single-thread Procs and (b) no cascading exits. **Phase 3+ MUST land a proc-table lock** before cascading rforks (userspace exec → fork → exec) become routine OR before multi-thread Procs land.
- **kernel-side TSan not feasible at v1.0**. The ROADMAP §5.2 TSan criterion is reinterpreted as "TLA+ scheduler.tla TLC under NoMissedWakeup + NoDoubleEnqueue + IPIOrdering + LatencyBound invariants." Should KCSAN-style kernel concurrency sanitization land post-v1.0, the empirical layer can be re-introduced; for now, the spec coverage is sufficient.
- **Scheduler test counter atomicity is now load-bearing** — proc_total_*/thread_total_*/pgrp_total_* read accessors return values via `__atomic_load_n`. Tests that compare counter deltas across SMP-driven workloads (work_stealing_smoke, future cross-CPU stress) will see sequentially-consistent reads modulo the RELAXED ordering. RELAXED is sufficient because counters carry no synchronization edge to other state.
