## What landed in this session window

### P4-Fix157: `userland_enter` SPSel discipline (`8172dfe` / `805e904` / `7d4711d`)

Trip-hazard #157 was reproduced + bisected at P3-H; deferred with a forward-looking P0 classification + F136 one-call guard. The R7 audit recorded "pre-eret state bit-identical between iter 1 and iter 2" and suspected QEMU virt simulation-state with `-cpu max` (MTE / branch predictor / GIC redistributor). All wrong — but the audit didn't capture PSTATE.SPSel or SP_EL1, which is exactly where the actual difference lived.

**Investigation arc**:

1. Wired the orphan reproducer (`kernel/test/test_userspace2.c`) into the test registry. Confirmed iter 1 PASS / iter 2 HANG within 30 seconds.
2. Ablated `-machine virt,mte=off`: still hung. **MTE ruled out.**
3. Added `tlbi vmalle1is + dsb ish + isb` immediately before eret: still hung. **Stale TLB / walker ruled out.**
4. Captured `qemu-system-aarch64 -d int -D trace.log`. Trace showed iter 2 in an infinite EL1→EL1 data abort loop:
   - ESR = 0x96000044 (EC=0x25 Data Abort current EL, IL=1, ISV=0, WnR=1, FSC=0x4 translation-fault L0).
   - FAR decreasing by exactly **0x120 per iteration**.
   - PC pinned at vector 0x200 entry.
   - SPSR = 0x23c5 (EL1h, all DAIF masked).
5. **0x120 = EXCEPTION_CTX_SIZE**. The FAR stride matched KERNEL_ENTRY's `sub sp, sp, #EXCEPTION_CTX_SIZE` + `stp x0, x1, [sp]` pattern. SP_EL1 was in unmapped territory and being decremented by 0x120 on every recursive abort entry.
6. Worked backward: SP_EL1 only gets written by `mov sp, X` at PSTATE.SPSel=1. The only such write in the user-launch path is `mov sp, x17` in `userland_enter`. Iter 1 was dispatched at SPSel=0 (parent's `wait_pid → sleep → sched` path). Iter 2 was dispatched at SPSel=1 (timer-IRQ-driven preempt of the parent's busy wait — `preempt_check_irq → sched → cpu_switch_context`, all at SPSel=1).

**Fix** (one logical step, three asm instructions added):

```asm
msr     SPSel, #0       // Force kernel-steady-state: sp aliases SP_EL0.
isb                     // Serialize the SPSel update before sp write.
mov     sp, x17         // SP_EL0 := user_sp (always; was buggy at SPSel=1).
```

**Effects**:
- At SPSel=0 entry: `msr SPSel, #0` is a no-op (already 0); behavior unchanged.
- At SPSel=1 entry: switches to SPSel=0; `mov sp` now correctly writes SP_EL0. SP_EL1 retains its prior value — under IRQ-driven dispatch that's `next_thread->ctx.sp` = the kthread's own kstack (valid kernel-side mapping; structurally correct exception stack for that thread).

**Regression test**: `kernel/test/test_userspace2.c` does two consecutive `rfork → exec_setup → userland_enter → wait_pid` rounds. In the new test-harness ordering (these tests run after the irqfwd suite which uses preempt_check_irq → sched), the parent is reliably at SPSel=1 by the time these tests execute; pre-fix BOTH iterations would hang deterministically in this ordering (vs. the historical "only iter 2 hangs" framing from the joey_run context where the parent was at SPSel=0).

**F136 (joey_run one-call guard)** retained for its structural role: `g_joey_elf_blob` is single-use BSS and a future Phase 5+ supervisor refactor that re-runs `joey_run` would race the build_init_elf zero-fill. Comment in `kernel/joey.c` updated to drop the now-stale #157-mitigation rationale.

**R8 audit** (focused adversarial review of `8172dfe`): 0 P0 / 0 P1 / 0 P2 + 4 P3 (F138-F141). Verdict: **fix sound**. F138 (stale `userland.S:25-35` comment block) + F139 (test_userspace2.c outdated header) closed in audit-close commit `7d4711d`. F140 (defensive `wfe; b 1b` loop is a diagnostic blind spot if eret falls through) + F141 (alternative `msr sp_el0, x17` would be simpler) deferred with explicit rationale (see `memory/audit_r8_closed_list.md`).

**Reference doc updates**:

### P4-H: VirtIO PCIe enumeration via ECAM (`9e73223` / `2d9a0ae`)

The PCIe-transport counterpart to P4-F's MMIO transport. Enumerates VirtIO devices reachable through QEMU virt's PCIe root complex (advertised in DTB as `compatible = "pci-host-ecam-generic"`, ECAM at PA `0x40_1000_0000` size 256 MiB).

**Public surface** (`<thylacine/virtio_pci.h>`):

```c
struct virtio_pci_dev {
    u8       bus, dev, fn;
    u8       header_type;
    u16      vendor_id;            // 0x1AF4
    u16      device_id;            // 0x1000..0x107F
    u16      virtio_device_id;     // VIRTIO_DEVICE_ID_* (block, net, rng, gpu, ...)
    u16      subsys_vendor, subsys_device;
    u8       class_code, subclass, prog_if;
    bool     is_modern;            // device_id in [0x1040, 0x107F]
    void    *cfg;                  // KVA of 4 KiB cfg space
};

void virtio_pci_init(void);
int  virtio_pci_dev_count(void);
struct virtio_pci_dev *virtio_pci_dev_get(int idx);
struct virtio_pci_dev *virtio_pci_find_by_device_id(u32 virtio_device_id);

u8  virtio_pci_cfg_read8 (const struct virtio_pci_dev *d, u32 off);
u16 virtio_pci_cfg_read16(const struct virtio_pci_dev *d, u32 off);
u32 virtio_pci_cfg_read32(const struct virtio_pci_dev *d, u32 off);
```

**Enumeration flow**:

1. `dtb_get_compat_reg("pci-host-ecam-generic", &ecam_pa, &ecam_size)`. Silent skip on absent root complex.
2. `mmu_map_mmio(ecam_pa, 1 MiB)` — maps **bus 0 only**. Full ECAM (256 MiB = 65536 pages) would exhaust the 2 MiB vmalloc L3; bus 0 mapping consumes 256 of 512 vmalloc pages (half the available L3 capacity, leaves ~80-page headroom after UART + GIC + 32 VirtIO MMIO slots).
3. For each dev 0..31: probe fn 0; if PCI_CFG_HEADER_TYPE bit 7 (multi-function) is set, probe fns 1..7.
4. For each populated function (vendor_id != 0xFFFF): if vendor 0x1AF4 + device_id in [0x1000, 0x107F], record into `g_virtio_pci_devs[VIRTIO_PCI_MAX_DEVS=16]`.
5. `virtio_device_id` derivation:
   - Modern (0x1040..0x107F): `device_id - 0x1040` (per VIRTIO 1.2 §4.1.2.1).
   - Legacy (0x1000..0x103F): `subsys_device` (per pre-1.0 spec).

**Bounds + alignment hygiene**: `cfg_read{8,16,32}` validate offset < 4096, naturally aligned, non-NULL device — return the bus-floating sentinel (0xFF / 0xFFFF / 0xFFFFFFFF) on violation rather than trapping.

**Boot wire-up**: `virtio_pci_init()` after `virtio_init()` in `boot_main`. Banner adds:

```
virtio_pci: ECAM at PA 0x4010000000 size 0x10000000 (3 VirtIO devices on bus 0, 32 functions probed)
```

**Test plumbing**: `tools/run-vm.sh` adds `-device virtio-rng-pci,id=rng_pci0` so the integration boot always has at least one PCI VirtIO device. Canonical run surfaces 3 VirtIO PCI devices on bus 0 (QEMU virt's default config plus the explicit rng_pci0).

**7 new tests**:
- `virtio_pci.init_called`               — count within [0, MAX_DEVS].
- `virtio_pci.count_within_bound`        — count >= 1 (the rng_pci0 attachment).
- `virtio_pci.devices_have_vendor`       — vendor_id == 0x1AF4; device_id in range; is_modern flag consistent.
- `virtio_pci.devices_have_cfg`          — d->cfg non-NULL; cfg_read16(VENDOR_ID) round-trip.
- `virtio_pci.find_rng`                  — find_by_device_id(VIRTIO_DEVICE_ID_RNG) hits.
- `virtio_pci.find_unknown_returns_null` — non-existent ID returns NULL.
- `virtio_pci.cfg_read_bounds`           — out-of-range / misaligned / NULL return absence sentinel.

**Reference doc**: `docs/reference/37-virtio_pci.md` (~135 lines).

**Deferred to first PCI-using driver chunk**:
- BAR mapping (drivers will read BARs via `cfg_read32(d, PCI_CFG_BAR0..5)` + map what they need).
- VirtIO PCI capability walk (`VIRTIO_PCI_CAP_*` per VIRTIO 1.2 §4.1.4) for common-cfg / notify / ISR / device-cfg region locations.
- INTx routing via DTB `interrupt-map` (`irqfwd` handles GIC INTIDs; the (bus,dev,fn,pin) → INTID walk is unwritten).
- MSI / MSI-X enable.
- PCIe bridges (Type 1 headers, secondary bus walking). QEMU virt's default has no bridges.

### P4-I (priority pick): virtio-blk userspace driver

The first userspace driver (Rust). Validates the Dev/Spoor/9P/handle plumbing end-to-end via a Rust process running at EL0 driving a VirtIO block device. Needs **sysroot + Rust cross-toolchain integration** — substantial scaffolding.

Likely sub-chunk decomposition:

1. **P4-Ia: sysroot scaffolding.** musl C runtime, headers, libc.a for `aarch64-unknown-linux-musl` (or a freestanding target). Build pipeline updates in `tools/build.sh`. Decide between musl-static / musl-dynamic / pure no_std Rust.
2. **P4-Ib: handle-table integration for KObj_MMIO + KObj_IRQ.** Per handoff 025 open follow-ups: "Handle-table integration for KOBJ_IRQ — held to P4-I+ alongside first userspace driver." A driver needs a handle to its device's MMIO range + a handle to its IRQ-blocker so userspace can wait for hardware events.
3. **P4-Ic: minimal Rust no_std virtio-blk crate.** Talks 9P over a handle to the kernel? Or directly drives the hardware via the MMIO handle? Plan 9 idiom would be the former, but at v1.0 P4-I may be simpler with the latter (driver-as-9P-server is P4-Id+).
4. **P4-Id: driver-as-9P-server.** Driver exposes `/dev/blk0` as a 9P endpoint via a server thread.

Note: P4-I can use either MMIO or PCI transport. QEMU exposes both flavors of virtio-blk; we have both substrates landed.

## Bug-hunting / lessons learned

**P4-Fix157 — what we learned**:

1. **Audit "state bit-identical" claims are exhaustive only against the registers they enumerate.** R7 checked TTBR0_EL1, SPSR_EL1, ELR_EL1, SCTLR_EL1, TPIDR_EL1, L0[0]. PSTATE.SPSel + SP_EL1 were not captured. Iteration-dependent behavior with "identical" state strongly hints at PSTATE / SPx-system-register involvement; next audit on EL transition code should explicitly dump SPSel + both SP regs.

2. **`-d int` traces with arithmetic-stride FAR are golden.** When the abort recurses with FAR decreasing by exactly EXCEPTION_CTX_SIZE per iteration, you've identified the recursion as KERNEL_ENTRY's `sub sp + stp`. That single number told us SP_EL1 was the corrupted register; everything else followed in 5 minutes.

3. **Ablations first, theories second.** R7's MTE / TLB / cache hypothesis chain consumed significant audit effort. A 30-second ablation (`-machine virt,mte=off`) ruled out MTE definitively. Same for `tlbi vmalle1is` before eret — 30 seconds proved the bug wasn't TLB-stale. Cheap ablations would have saved R7 from publishing the QEMU-virt-simulation-state hypothesis.

4. **The "P2-Cc kernel runs at SPSel=0" assumption was load-bearing and unenforced.** It held under specific dispatch patterns (sleep/wakeup) but failed under IRQ-driven preempt. Any code path that depends on an unenforced PSTATE assumption is a latent #157.

**P4-H — what we learned**:

1. **Vmalloc is tight.** 256 of 512 pages consumed by mapping just bus 0. Full ECAM (256 MiB) would have required either an L2 block mapping or multi-L3 vmalloc. Plan future MMIO-heavy chunks (Phase 5+ networking, Stratum buffers) against this constraint.

2. **DTB-driven enumeration scales well.** `dtb_get_compat_reg("pci-host-ecam-generic", ...)` already returned the ECAM range correctly because the existing 2-cell address / 2-cell size parser matches the QEMU virt convention. No DTB-side changes were needed.

3. **`tools/run-vm.sh` device attachments matter for test reliability.** Adding `-device virtio-rng-pci,id=rng_pci0` makes the test `virtio_pci.find_rng` deterministic. The 3 VirtIO PCI devices we surface in the canonical run include this RNG plus whatever QEMU virt's default config exposes (possibly a balloon device + GPU placeholder; not yet investigated).
