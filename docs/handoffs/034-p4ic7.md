## Three surprises documented for future sessions

### 1. Kernel L3-mapping constraint is image + firmware ≤ 2 MiB, not image alone

`arch/arm64/kernel.ld` asserts `(_kernel_end − _kernel_start) < 0x200000` (image alone < 2 MiB). The **actual** runtime constraint is image + firmware-offset ≤ 2 MiB — `arch/arm64/mmu.c::mmu_map_kernel` maps exactly one 2 MiB L2 block at the kernel image's load address, with the firmware-area pages (typically 512 KiB on QEMU virt) marked invalid in the L3 table. With firmware offset 512 KiB, usable kernel image = 2 MiB − 512 KiB = 1536 KiB. The assert in kernel.ld is **conservative-low** (image-only); a runtime fault happens when image alone is < 2 MiB but image + firmware exceeds 2 MiB. P4-Ic7 hit this when adding a 128-KiB blob to baseline 1440 KiB image — pushed total to 1568 KiB image_size = 2080 KiB image+firmware > 2 MiB. Boot silently faulted during .bss clearing in `start.S` past the L3-mapped region. Resolution: shrunk three existing 256-KiB ELF blobs to 128 KiB each (saves 384 KiB; image_size now 1188 KiB; 348 KiB headroom).

**Future audit suggestion**: tighten the kernel.ld assert to also account for the firmware offset. The current 0x200000 check missed by 32 KiB.

### 2. SYS_PUTS dereferences user pointers in kernel mode without uaccess-fault recovery

`kernel/syscall.c::sys_puts_handler` reads the user-VA buffer directly: `const char *buf = (const char *)(uintptr_t)buf_va; for (...) uart_putc(buf[i]);`. The v1.0 kernel has no uaccess-emulation: a kernel-mode dereference of an unmapped user VA produces an **UNHANDLED translation fault** (not a recoverable copy_from_user-style error path). For binaries whose .rodata fits in page 0 (probes, hello-rs), this never triggers because page 0 is always touched by execution. For P4-Ic7's binary (~74 KiB → .rodata spans page 1 too), accessing a string literal at user-VA 0x401XXX from kernel mode hits this hazard.

**Workaround in P4-Ic7**: `pretouch_rodata_pages()` reads page 1 of the LOAD segment at user-mode entry, forcing `userland_demand_page` to map it before any SYS_PUTS. If a future binary grows past 0x402000, the pretouch loop needs extension.

**Future audit suggestion (Phase 5+)**: add proper uaccess-emulation to all syscall handlers that dereference user pointers. Either copy_from_user-style fault-recovery (kernel sets up a fixup landing pad before the dereference, faults route to it) or pre-flight VMA-walk-and-fault-in (walks user pgtable, demand-pages missing entries before the dereference).

### 3. Disk-image idempotency: the test must not mutate its read-source region

First attempt at P4-Ic7 used overlapping read + write regions (read sectors [1..N+1), write the same sectors [1..N+1)). Pass A reads pattern A from disk (initial bake-in by mkdisk.py); pass B overwrites with pattern B; pass C reads + verifies B. Test passes on first run. **But the disk image is mutated** — pattern B now lives in sectors [1..N+1) on the host file backing virtio-blk. Subsequent test runs: pass A reads pattern B, mismatches expected pattern A, FAIL.

Resolution: non-overlapping regions. READ region [1..N+1) (pattern A, **never modified**); WRITE region [N+1..2N+1) (pattern B written + verified). Default disk size auto-scales pass_bytes to fit; the strict 1 GiB R + 1 GiB W exit criterion requires 2 GiB disk (`THYLACINE_DISK_SIZE=2G`).

**Future test-discipline rule**: any test that issues writes to a backing store MUST EITHER (a) restore the pre-state on success, OR (b) write to a region disjoint from any region it reads. (b) is preferred because (a) is fragile under abnormal exits.

## Common pitfalls for future sessions

- **Don't grow kernel .bss past ~350 KiB without checking image_size**. Run `python3 -c "import struct; f=open('build/kernel/thylacine.bin','rb'); f.seek(0x10); print(struct.unpack('<Q', f.read(8))[0]//1024)"` and ensure image_size + 512 KiB firmware ≤ 2048 KiB. If close to the limit, audit existing test blobs for sizing slack.
- **Don't add userspace .rodata past page 1 without extending `pretouch_rodata_pages()`**. The pretouch loop currently covers page 1 (0x401000) only; a binary spanning 0x402000+ needs another touch.
- **Test-disk regeneration**: `./tools/build.sh disk` regenerates with the current `THYLACINE_DISK_SIZE`. If you ran the 2 GiB exit-criterion test and want to return to default, run `./tools/build.sh disk` (no env) to get back to 16 MiB.
