## What landed since handoff 014

### P3-A — F75 close (this commit)

`9d5c271` substantive. Hash-fixup follows.

**What it fixes**: R5-H F75 (cascading-exits SMP UAF). Parent A in exits() runs proc_reparent_children walking A's children to rewrite c->parent = kproc. If a child B is concurrently in its own exits() reading p->parent (= A) for the wakeup target, and B's wakeup line fires after A's full exits → ZOMBIE → wakeup-grandparent → grandparent reaps A → A freed chain, B accesses freed memory. The new `g_proc_table_lock` serializes A's mutation with B's read; the wakeup-INSIDE-lock structure additionally prevents A from being reaped between B's read and B's wakeup.

**Lock discipline established**:

- `g_proc_table_lock` (global spinlock; spin_lock_irqsave for IRQ-mask discipline forward-compat).
- Protects: each Proc's children list head + per-child sibling chain, parent pointers, ALIVE→ZOMBIE state transitions, exit_msg / exit_status, companion Thread's THREAD_EXITING transition.
- **Lock order**: `proc_table_lock → r->lock` (single direction; established by `exits()` calling `wakeup(parent->child_done)` INSIDE the lock).
- **Forbidden order**: `r->lock → proc_table_lock`. Would deadlock against the established order.
- `wait_pid_cond()` is the only `r->lock`-holder that touches Proc-lineage state; it deliberately reads the children list WITHOUT `proc_table_lock`, relying on three v1.0 invariants:
  1. Single-thread Procs → parent's children list is single-writer (the parent's own thread).
  2. Per-child state visibility via wakeup→sleep release/acquire on r->lock.
  3. First-call tolerance: stale state at sleep entry is OK because the next wakeup re-evaluates.
- **Phase 5+ MUST refactor**: when multi-thread Procs land, invariant (1) weakens. wait_pid_cond will need to acquire proc_table_lock, AND the sleep protocol must be refactored to break the resulting r->lock → proc_table_lock cycle. Documented in `kernel/proc.c::g_proc_table_lock`'s header comment + `docs/reference/14-process-model.md` + `docs/phase3-status.md` trip-hazard #101.

**Self-audit found** (and fixed in same chunk):

- **Wakeup-inside-lock requirement**: an earlier draft released the lock before the wakeup, leaving a window where the parent could be reaped between release and wakeup → UAF. Caught during self-audit by walking the timeline carefully. Fix: keep wakeup INSIDE the lock; the lock-order proc_table_lock → r->lock is established here.
- **`g_next_pid` non-atomic**: cascading rforks under SMP can have multiple Procs in proc_alloc concurrently from different CPUs; non-atomic `++` would let two collide on the same PID. Fixed to `__atomic_fetch_add(&g_next_pid, 1, RELAXED)`.

**New tests** (49/49 PASS, +2 from 47):

- `proc.cascading_rfork_wait_smoke` — single iteration of 3-level lineage: boot → child → grandchild. boot rforks A; A rforks B in its entry; B exits; A waits for B (reaps); A exits; boot waits for A (reaps). Counter checks (proc + thread totals) for leaks.
- `proc.cascading_rfork_stress` — 100 iterations of the same pattern. Regression coverage: at 100 iters × 3 levels × 4 vCPUs work-stealing, real chance of triggering the F75 race window if the lock is removed in a future change.

**Reference doc updates**:

- `docs/reference/14-process-model.md` extended with the proc-table-lock discipline section (protected fields, lock ordering, soundness argument for wait_pid_cond's lockless walk, Phase 5+ trip-hazard).
- `docs/phase3-status.md` created as the Phase 3 pickup guide.

## Trip-hazards added at P3-A

100. **Lock order: `proc_table_lock → r->lock`** (P3-A). Established in `exits()` to keep the wakeup-target Proc alive through the wakeup. Reverse forbidden — would deadlock.
101. **`wait_pid_cond` lockless walk REQUIRES single-thread Procs** (P3-A). Phase 5+ MUST refactor: wait_pid_cond must acquire proc_table_lock, AND the sleep protocol must be refactored to break the resulting r->lock → proc_table_lock cycle.
102. **`g_next_pid` is atomic** (P3-A). `__atomic_fetch_add(_, 1, RELAXED)`. Required for SMP cascading rforks where multiple Procs in proc_alloc from different CPUs would otherwise collide.
103. **Cascading-rfork tests exercise the lock paths** (P3-A). `proc.cascading_rfork_wait_smoke` + `proc.cascading_rfork_stress` regression-protect against future changes that bypass the locking discipline.
