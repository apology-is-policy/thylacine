### P2-Fb — vmo.tla spec

`ce058cc` substantive + `ceca889` hash fixup.

- New spec `specs/vmo.tla` (~290 LOC TLA+) + 4 cfg files (1 clean + 3 buggy).
- Models the dual-refcount lifecycle for VMOs per ARCH §19. **Pins ARCH §28 I-7 (NoUseAfterFree)**.
- Per-VMO state: `handle_count` (open handles) + `mapping_count` (open mappings) + `pages_alive` (membership flag).
- 5 spec actions: VmoCreate, HandleOpen, HandleClose, MapVmo, UnmapVmo. 3 buggy variants:
  - BUGGY_FREE_ON_HANDLE_CLOSE: premature free at handle_count=0 ignoring mapping_count.
  - BUGGY_FREE_ON_UNMAP: symmetric.
  - BUGGY_NEVER_FREE: delayed free (counts=0 but pages_alive=TRUE).
- The `NoUseAfterFree` invariant uses **iff form** to catch BOTH directions:
  - LHS true RHS false → premature (UAF).
  - LHS false RHS true → delayed (leak).
- MaxRefs=2 bounds state space; 100 distinct states / depth 9 / <1s.

## Trip-hazards added since handoff 011

### P2-Fa (handles.tla spec)

- **Buggy-config matrix orthogonal flags** (BUGGY_ELEVATE / BUGGY_HW_TRANSFER / BUGGY_DIRECT_TRANSFER / BUGGY_CAPS_ELEVATE): each flag enables one buggy action. Combinations not tested; if a future change makes two buggy actions interact, add a combined buggy.cfg.
- **Per-handle `via` provenance is a tag, not a runtime state machine**: the spec models 4 provenance tags as enum values; the impl doesn't track provenance per handle. The asymmetry is fine for now (the spec catches buggy "direct" via state inspection; the impl prevents it via absence-of-syscall) but if the impl were ever to track provenance, the spec mapping needs to add it.

### P2-Fb (vmo.tla spec)

- **MaxRefs=2 bounds state space artificially**: the spec proves NoUseAfterFree at refcount up to 2. Higher refcounts are inductively-implied-correct but not exhaustively explored. If a future change introduces a refcount-dependent bug class (e.g., refcount overflow), the spec needs MaxRefs increase + re-verification.
- **Iff-form invariant requires both directions**: NoUseAfterFree iff catches premature AND delayed free. A future "implies-only" weakening (LHS → RHS only) would catch one direction; the iff is load-bearing.

### P2-Fc (kernel handle table impl)

- **PROC_HANDLE_MAX = 64**: container init flows that hold > 64 handles hit -1 (table full). Phase 5+ refactors to growable RB-tree.
- **No underlying-kobj refcount integration except KOBJ_VMO**: handle_close / handle_table_free zero the slot for non-VMO kinds without decrementing any refcount. Future kinds (KOBJ_CHAN at Phase 4; KOBJ_PROCESS / THREAD / MMIO / IRQ / DMA / INTERRUPT later) need their refs wired into `handle_release_obj` + `handle_acquire_obj` switches.
- **Single-CPU lifecycle**: handle_alloc / close / dup / get are not internally synchronized. Phase 5+ adds a per-Proc handle-table lock when SMP syscalls go live.
- **Defensive runtime check on handle_alloc(KOBJ_VMO) added at R5-F**: if the convention is broken (caller passes a Vmo with handle_count=0), handle_alloc extincts. This is forward-looking enforcement for Phase 4.

### P2-Fd (kernel VMO impl)

- **Eager allocation via order rounding**: a 5-page VMO allocates 8 pages (order 3); waste of 3 pages. Acceptable for v1.0.
- **After unref/unmap that frees the VMO, the v pointer is invalid**: tests use cumulative counters (vmo_total_destroyed) rather than dereferencing freed pointers.
- **vmo_map / vmo_unmap are refcount-only at v1.0**: Phase 3 wires VMA + page-fault handler integration.
- **Anonymous VMOs only**: VMO_TYPE_PHYS at Phase 3; VMO_TYPE_FILE post-v1.0.
- **Single-CPU lifecycle**: Phase 5+ adds atomic ops on handle_count + mapping_count.
- **F-SF1 size overflow guard**: caller-side OOM via passing huge sizes is rejected. Removing this guard re-introduces F-SF1 + makes F59 (order_for_pages infinite loop) reachable.

### R5-F audit close

- **Cache flag drop (PANIC_ON_FAIL)**: g_handle_table_cache + g_vmo_cache no longer panic on alloc OOM; they return NULL. Callers must handle the NULL return. The proc_alloc rollback path is now reachable for the FIRST time. The pgrp_alloc + proc_alloc paths still use PANIC_ON_FAIL on g_pgrp_cache + g_proc_cache (out-of-scope for R5-F; could be a future cleanup).
- **proc_alloc counter discipline**: g_proc_created++ now hoisted to BEFORE any failure path. This is a subtle ordering change — the counter increments even if the eventual rfork OOM-rollback fires. proc_free does balance with destroyed++. If a future code path adds a NEW failure mode in proc_alloc that doesn't go through proc_free, the counters would drift.
- **Handle release/acquire default arms**: out-of-enum kind values now extinct. If memory corruption produces a kind value that happens to fall through one of the case labels, the default arm is bypassed. Defense-in-depth is best-effort.
