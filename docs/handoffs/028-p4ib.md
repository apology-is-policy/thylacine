# Handoff 028 — P4-Ib (hardware-handle integration + capability gating, spec-first, R9 audit closed)

**Tip**: `957284c` (P4-Ib hash fixup). Predecessor: `f23ab34` (handoff 027).

## TL;DR

P4-Ib delivers the **hardware-handle creation substrate** that the first userspace driver (P4-Ic — virtio-blk) builds atop. Three new TLC-verified invariants in `specs/handles.tla` pin the semantics: **NoHwDup** (no `handle_dup` of hw kinds), **HwResourceExclusive** (at most one alive handle per hw kobj across all procs), and **HwHandleImpliesCap** (every hw-handle holder has `CAP_HW_CREATE`). The impl maps cleanly to the spec: new `caps_t caps` field on `struct Proc`; new `struct KObj_MMIO` with a global PA-claim table; INTID claim tracking added to `kernel/irqfwd.c`; three new syscalls (`SYS_MMIO_CREATE`, `SYS_IRQ_CREATE`, `SYS_IRQ_WAIT`) with capability check + arg validation + rollback.

**R9 adversarial audit** prosecuted the chunk with `general-purpose` agent — surfaced **12 findings (F142-F153)**: 1 P0 + 3 P1 + 3 P2 + 5 P3. The headline was **F142 (P0)**: `SYS_IRQ_CREATE` could let a userspace caller with `CAP_HW_CREATE` seize the kernel's timer (INTID 30) or IPI (SGI 0) because `gic_attach` silently overwrites and those INTIDs weren't in `g_intid_claimed[]`. Fix: `irqfwd_init()` pre-marks kernel-owned INTIDs + the syscall layer rejects `intid < 32` (SPI-only policy). **10 findings closed in the audit-close commit**; **2 P3 deferred** with explicit forward-looking rationale.

**Phase 4 hardware-handle substrate complete**: native drivers can now claim MMIO ranges + IRQ subscriptions through the kernel handle table with all the spec invariants enforced. P4-Ic implements `kobj_mmio_map_into_user` (deferred from P4-Ib because it needs Burrow-type extension for MMIO pages) + the first real userspace driver. **Test count: 180 → 196**.

**Investigation arc — F142 (the P0)**:

The audit agent's prosecution chain:
1. `gic_attach` (in `arch/arm64/gic.c`) silently overwrites any existing handler at the given INTID — no double-attach check.
2. Kernel-internal callers (timer at INTID 30, IPI_RESCHED at SGI 0) attach via `gic_attach` directly, bypassing `kobj_irq_create` and `intid_try_claim`. So `g_intid_claimed[30] == false` even though the kernel timer owns it.
3. A userspace driver proc holding `CAP_HW_CREATE` calls `t_irq_create(intid=30, RIGHT_SIGNAL)`.
4. Cap check passes; arg validation passes; `kobj_irq_create(30)` is invoked.
5. `intid_try_claim(30)` sees claim is false → flips true → returns true.
6. `gic_attach(30, kobj_irq_dispatch, k)` overwrites `g_handlers[30].handler` from `timer_irq_handler` to `kobj_irq_dispatch`.
7. Every subsequent timer fire now wakes the rogue `KObj_IRQ` instead of running the kernel's tick handler. Scheduler tick freezes → I-8 violation.

**Fix mechanism**: `irqfwd_init()` (called from `boot_main` after `virtio_init`) explicitly sets `g_intid_claimed[IPI_RESCHED] = g_intid_claimed[TIMER_INTID_EL1_PHYS_NS] = true`. Boot banner now prints `irqfwd: reserved kernel INTIDs 0,30`. Then `sys_irq_create_handler` rejects `intid < 32` — explicit SPI-only policy at the syscall boundary. New regression test `handle_hw.irq_kernel_reserved_rejected` verifies kernel-context `kobj_irq_create` against the reserved INTIDs.

1. **Implement the deferred `kobj_mmio_map_into_user` syscall** (the only syscall not delivered at P4-Ib). ## Bug-hunting / lessons learned

**P4-Ib — what we learned**:

1. **Spec-first paid off concretely.** The spec extension was relatively quick to write (the existing handles.tla model was extensive); TLC verification surfaced one bug in HwResourceExclusive's set-comprehension syntax (pattern-binding `<<p, h>>` doesn't unpack tuples in TLC's set iteration as I'd written it). Fixed immediately at the spec level — cost was minutes vs the impl-side equivalent being a runtime invariant violation that might escape testing.

2. **The R9 P0 was discoverable only by an adversarial reviewer.** Internal tests all passed because none of them attempted to create a KObj_IRQ on a kernel-reserved INTID. The audit prosecutor's framing ("what could a userspace caller with CAP_HW_CREATE do?") immediately surfaced the path: try `intid=30`. None of my impl-side checks caught it because `gic_attach`'s silent overwrite is at a layer below `intid_try_claim`. Lesson: **whenever an existing kernel-internal API has its own state that the new abstraction depends on, the abstraction MUST be the only writer or MUST take over the state's initial population.** My `g_intid_claimed[]` assumed it owned the claim namespace; actually `g_handlers[]` was already populated by kernel callers. Closing the gap required `irqfwd_init` to seed the claim table from the pre-existing handler installations.

3. **F148 (non-atomic refcounts) was a "v1.0 single-thread-Procs narrows the exposure" hazard.** Today's tests don't exercise the SMP race; UBSan doesn't catch it. The audit caught it via code review — `__atomic_fetch_*` was missing where it was needed structurally. The fix is mechanical (~10 lines across mmio_handle.c + irqfwd.c) but it's exactly the kind of latent bug that explodes the first time P4-Ic actually has a driver proc using its handle from a different CPU than the closer. **Atomic ops in refcount lifecycles are not optional even at v1.0 single-thread-Procs — the lifecycle racetrack opens with the first cross-CPU close.**

4. **F143 (UAF across blocking sleep) is the canonical "borrow a ref" oversight.** Standard pattern in Linux kernel code: any handle-table lookup whose pointer escapes the lookup must bump the underlying refcount. P4-Ib's `sys_irq_wait_handler` had the lookup but missed the bump. Fix is one-line `kobj_irq_ref(k)` before + `kobj_irq_unref(k)` after the blocking call. This pattern will be replicated for every Phase 5+ syscall that holds a handle pointer across a blocking operation (poll, futex, recvmsg, etc.) — worth adding to the syscall-writing pattern docs.

5. **Spec coverage gaps live where the impl is *more strict* than the spec.** F153 was the spec missing the same-proc-overlap case because impl's global `g_mmio_claims` rejected it implicitly. The spec wasn't *wrong* — it just didn't exercise the case. Lesson: when an impl is more strict than the spec, the spec should be relaxed to cover the broader case so TLC explores it; the invariant catches both.

6. **Capability gating is structural, not behavioral.** `HwHandleImpliesCap` is a state invariant — "if you hold the handle, you have the cap." It pins both the alloc-time check AND the persistence-of-cap invariant (via ReduceCaps's restriction). The impl alone (cap check at create) only pins half; without `ReduceCaps`-of-`CapHwCreate` interlock the state could drift (cap dropped post-alloc → handle still held → spec invariant violated). Documented as F150 deferred — the Phase 5+ implementer of cap-drop has the implementer note in `caps.h` flagging the required -EBUSY check.
