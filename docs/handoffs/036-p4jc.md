# Handoff 036 — P4-Jc (virtio-net-loop: steady-state RX recycling)

**Tip**: `7ac0f2c` (P4-Jc hash fixup) on `main`. **virtio-net steady-state operational; foundation for 9P-over-TCP laid.**

This window lands the steady-state virtio-net userspace driver in a single sub-chunk that closes the gap left by P4-Jb's single-shot probe:

- **P4-Jc** (`dbf1425` substantive / `7ac0f2c` hash fixup) — virtio-net-loop: RX descriptor recycling + TX descriptor reuse via static `descriptor[k] → buffer[k]` mapping. 24 ARP round-trips through 16-descriptor TX+RX rings on each side. **Wraparound proven on both directions.**

This is what 9P-over-TCP in Phase 5 (= ROADMAP §7 Stratum integration) sits on. The descriptor-recycling discipline established here is the foundation for any continuous-RX driver Thylacine ships.

---

## What changed at the device layer

Nothing. Same composed hw-handle SVC surface as virtio-net-arp (MMIO bank + IRQ + 2× DMA). No kernel changes. The chunk is pure userspace + a bundled blob-cap shrink.

The interesting work is in `usr/virtio-net-loop/src/main.rs`:

### RX recycling — `drain_rx_used_with_recycle`

```
for each new used.ring entry:
    parse + validate the frame
    re-publish desc_id back to avail.ring[rx_avail_idx % QUEUE_SIZE]
    rx_avail_idx = wrapping_add(1)
dsb; write avail.idx; dsb; QueueNotify(RX)
```

Without this, the RX queue runs dry after the 16 pre-published descriptors are consumed and packet delivery stalls.

### TX reuse — static `descriptor[k] → buffer[k]` mapping

```
populate_tx (at init):
    for k in 0..16:
        build TX buffer at offset 0x700 + k * 64 (=
            [virtio_net_hdr (12)][Eth (14)][ARP (28)] = 54 B)
        descriptor[k]: addr = buf k, len = 54, no NEXT
        avail.ring[k] = k

send_pending_tx (each iteration):
    while tx_sent < N_TARGET
       && tx_avail_idx.wrapping_sub(tx_seen_used) < QUEUE_SIZE:
        tx_avail_idx = wrapping_add(1)
    if any: dsb; write avail.idx; dsb; QueueNotify(TX)
```

The device cycles through descriptors as we advance avail.idx. Wrap past QUEUE_SIZE re-uses descriptor 0, safe because the in-flight guard prevents reuse while the device still owns it. **Buffers are written once at init and never modified again** — TX direction is read-only from the device's perspective.

### Steady-state loop

```
send_pending_tx (initial batch)
for _iter in 0..MAX_IRQ_ITERATIONS:
    if rx_validated >= N_TARGET: break
    irq_wait()
    read+ack InterruptStatus
    drain_tx_used    (advances tx_seen_used)
    drain_rx_used + recycle  (advances rx_seen_used + rx_avail_idx)
    send_pending_tx  (advances tx_avail_idx + kicks if changed)
```

In practice the 24 round-trips complete in 1-2 IRQ wakes per direction (slirp responds in tens of microseconds; IRQ coalescing batches multiple completions per wake).

## Volume choice — why N = 24

- N > QUEUE_SIZE = 16 is the minimum that exercises wraparound on both queues.
- N = 1.5× QUEUE_SIZE = 24 is the smallest value that meaningfully demonstrates recycling without making the test slow. Larger values would burn boot-time budget without adding signal.
- Each ARP request gets one slirp reply, so 24 requests → 24 replies; counter hits target deterministically.

The boot log shows `virtio-net-loop: PASS — 24/24 ARP replies validated; tx_completed=24; RX desc recycled past QUEUE_SIZE`.

## Bundled blob-cap shrink (image_size discipline)

Adding a 128 KiB blob for virtio-net-loop would have pushed image_size from 1448 → 1576 KiB. Combined with the 512 KiB firmware area, total = 2088 KiB > 2 MiB. **Image would have silently faulted during .bss clearing past the L3-mapped region**, exactly the trap P4-Ic7 hit and handoff 034 documented as deferred-audit item R12-bss-2mib.

Solution: shrink all 8 test-binary blobs uniformly from 128 KiB → 96 KiB.

- Largest binary today: virtio-net-arp at 76 KiB → 20 KiB headroom per blob.
- Total savings: 8 × 32 KiB = 256 KiB.
- New image_size: 1352 KiB (net −96 KiB after adding the new ~96 KiB blob).
- Total: 1352 + 512 = 1864 KiB; 184 KiB headroom under 2 MiB.

If a future binary exceeds 96 KiB, the cap shrinks again OR the binary gets size-tuned (release mode opt-z is already on). Either way, the kernel.ld assert at `< 0x200000` is conservative-low — it's the runtime constraint (image + firmware) that matters, and R12-bss-2mib (tighten the assert to account for firmware offset) remains a Phase 5+ quality-of-life item.

## u16 wraparound discipline (4 counters)

All four monotonic counters use the same idiom:

```rust
let mut idx: u16 = 0;
// ...
idx = idx.wrapping_add(1);  // wraps at 65536
while seen != cur_used {      // != is correct under wrap
    // ...
    seen = seen.wrapping_add(1);
}
// in-flight count:
let inflight = idx.wrapping_sub(seen);  // also wraps correctly
```

This is the correct VIRTIO spec idiom. `<` comparison would break under wrap; `!=` is wrap-safe. The pattern matches what P4-Jb established for rx_seen_idx and extends to all four counters here.

For N = 24, no counter actually wraps in the steady-state test (max value = 24, far from 65536). But the discipline is correct under hypothetical heavy use.

## pretouch_rodata_pages discipline

LOAD spans pages 0+1 (filesz = 0x136e = 4974 B; objdump-confirmed). Page 2 is NOT in the LOAD segment — touching it would itself fault. Therefore pretouch covers page 1 only.

Rule of thumb (per handoff 035): `pretouch covers page k iff (LOAD_top > k * PAGE_SIZE)`.

If a future expansion pushes the LOAD past 0x402000, the pretouch must extend in lockstep. Until then, page-1-only is correct.

- **Error-path**: every failure returns `Err(())` or calls `t_exits(1)`.
- **Idempotency**: probe runs once per process.

- **Compile-time invariants**: `const _: () = { assert!(...) }` pins TX pool fits in ring DMA + per-slot stride covers frame + virtio_net_hdr.
- **Boundary conditions**:
  - QUEUE_SIZE 16 ≤ QUEUE_NUM_MAX (QEMU default 256).
  - N_TARGET 24 > QUEUE_SIZE 16 — proves recycle on both sides.
  - MAX_IRQ_ITERATIONS 256 defensively bounds the loop.
  - u16-wraparound discipline on all 4 counters (tx_avail_idx, tx_seen_used, rx_avail_idx, rx_seen_used).
  - desc_id read from used.ring is u32 → cast to u16 safe (always ∈ [0..QUEUE_SIZE) by VIRTIO flow control).
  - **Used-ring slot reuse correctness**: VIRTIO guarantees the device cannot advance used.idx beyond what the driver has acknowledged + what's been re-published to avail. So `used.idx − seen ≤ QUEUE_SIZE` always, and the slot-index expression `seen % QUEUE_SIZE` never reads a stale entry from a previous wrap.

- **RX kick on every drain batch** — currently unconditional. Some VIRTIO devices don't strictly require RX kicks (they poll avail as needed). A future micro-optimization could point all 16 descriptors at one buffer; trades 960 B saved for added device-coherency-on-shared-read assumption (safe on QEMU virt; might bite on real hardware). Not worth doing until profiling shows it matters.

## Common pitfalls for future sessions

- **Image+firmware ≤ 2 MiB**. Adding a new test binary means adding a new blob; check that 8 × blob_max + kernel image + 512 KiB firmware stays under 2 MiB. The kernel.ld assert is conservative-low (image alone < 2 MiB); the real constraint is image + firmware ≤ 2 MiB. If image_size grows past ~1500 KiB, consider shrinking the blob caps.
- **pretouch_rodata_pages MUST match LOAD bounds.** Both directions: covering pages outside LOAD faults; missing a page inside LOAD breaks SYS_PUTS on its strings. Check with objdump after each substantive code change to the binary.
- **VIRTIO descriptor layout strictness**. Each descriptor is exactly 16 bytes: `le64 addr; le32 len; le16 flags; le16 next`. Off-by-one corrupts the chain.
- **VIRTIO avail/used ring stride**. avail.ring[k] is le16 (2 B); used.ring[k] is `{le32 id; le32 len}` (8 B). Don't confuse them.
- **VIRTIO u16 idx wraparound**. Use `wrapping_add(1)` + `!=` comparison; `<` is wrong.
- **VIRTIO flow control invariant**. The device cannot advance used.idx by more than QUEUE_SIZE beyond the driver's last-seen index — the avail ring's contents bound what the device can consume. This is what makes used-ring slot reuse safe.
