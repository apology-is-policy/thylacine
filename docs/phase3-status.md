# Phase 3 — status and pickup guide

Authoritative pickup guide for **Phase 3: Address spaces + page-fault handler + per-Proc TTBR0 + exec/userspace bringup**.

## TL;DR

Phase 3 lifts the kernel from "kernel-only multitasking" to "userspace processes running in EL0." Major deliverables: per-Proc TTBR0 (independent address spaces), page-fault handler (demand-page allocation, COW, stack growth), VMO mapping infrastructure (mmap_handle / vmo_map syscall surface), exec syscall (parse-validate-map-dispatch from P2-Ga's loader), minimal `/init` userspace process. Phase 4's Stratum 9P integration depends on Phase 3's address-space work landing first.

Per `ROADMAP.md §6`. Phase 3 entry chunk **P3-A** (R5-H F75 proc-table lock close) has landed early to establish the SMP locking discipline before cascading-rfork-based userspace patterns layer on top.

### NEW at P3-Ec

### NEW at P3-Eb

147. **`exec_setup` requires p->vmas == NULL** (P3-Eb). v1.0 doesn't replace-in-place. Calling `exec_setup` on a Proc with existing VMAs returns -1. Phase 5+ exec(2) syscall handler will tear down the existing image atomically (with rollback on failure) before calling exec_setup.

148. **Page-aligned segments only** (P3-Eb). `exec_setup` rejects segments where `vaddr` or `file_offset` low 12 bits are non-zero. ELF spec permits `vaddr ≡ offset (mod p_align)` with non-zero low bits. v1.0 limitation; toolchain output (clang, gcc) page-aligns by default.

149. **Per-segment VMO lifecycle: vmo_unref AFTER vmo_map** (P3-Eb). `exec_setup` calls `vmo_unref` (drop caller-held handle_count) AFTER `vmo_map` succeeds. The mapping_count taken by `vma_alloc` keeps the VMO alive until proc_free's vma_drain. Without the vmo_unref, handle_count would stay at 1 forever (no handle table entry to close), and the VMO would leak.

150. **User stack at fixed VA**: `[EXEC_USER_STACK_BASE, EXEC_USER_STACK_TOP) = [0x7FFFC000, 0x80000000)` (P3-Eb). 16 KiB. Phase 5+ adds growable stack via demand-grow on faults below stack base, plus per-Proc stack VA randomization (stack ASLR).

151. **Caller responsible for partial-state cleanup on `exec_setup` failure** (P3-Eb). On non-zero return, the Proc is in a partial state with some VMAs installed and some not. Caller calls `proc_free` with `state=ZOMBIE` to clean up; vma_drain releases whatever was installed.

### NEW at P3-Dc

136. **`mmu_install_user_pte` is NOT serialized at v1.0** (P3-Dc). Single-thread Procs from P2-D mean only the running thread of Proc P faults P's pgtable. Phase 5+ multi-thread Procs MUST add a per-Proc pgtable lock around the walk-allocate-install sequence. Without it: two CPUs concurrently fault on the same Proc and both walk an L0 entry that's invalid; both call alloc_pages for L1; both install table descriptors. The second install overwrites the first → the first L1 leaks (becomes unreachable mid-walk; proc_pgtable_destroy never finds it). Documented as the single largest Phase 5+ refactor for the demand-paging path.

137. **PTE encoding for user pages**: `AP_RW_ANY` (RW EL0+EL1) for VMA_PROT_WRITE else `AP_RO_ANY` (RO EL0+EL1); `UXN` clear for VMA_PROT_EXEC else set; `PXN` unconditionally (kernel never executes user pages); `AF + nG + SH_INNER + ATTR_IDX(NORMAL_WB)`. The defensive W+X reject in mmu_install_user_pte mirrors vma_alloc's reject and ELF loader's reject — three layers of I-12 enforcement.

138. **Demand-paged backing PA = `page_to_pa(vmo->pages) + (vmo_byte_off & ~PAGE_MASK)`** (P3-Dc). v1.0 anonymous VMOs allocate pages eagerly via `alloc_pages(order)`; pages are contiguous starting at `vmo->pages`. The arithmetic resolves the VMO byte offset to the corresponding PA. Phase 5+ VMO_TYPE_FILE (Stratum-backed) will need a per-page lookup (a more general `vmo_get_page(vmo, byte_off) → struct page *` interface).

139. **`userland_demand_page` is exposed in fault.h for tests** (P3-Dc). The function is callable from kernel tests with a synthetic fault_info + manually-constructed Proc, bypassing the need to trigger a real EL0 fault (impossible at v1.0 pre-exec). When P3-E lands the EL0 sync exception vector, the dispatcher path becomes the "hot" caller; the test path remains for unit testing.

140. **mmu_install_user_pte is idempotent on identical re-install** (P3-Dc). Returns 0 without allocating a new page if the existing L3 entry has matching PA + prot bits. Mismatching install (different PA at same vaddr) returns -1 — signals a logic bug in the demand-paging path. Phase 5+ COW will need a different protocol (atomic compare-and-swap on the L3 entry under the per-Proc pgtable lock).

141. **Per-Proc pgtable walker reads PTE bits via `pa_to_kva`** (P3-Dc). Both the install path (mmu_install_user_pte) and the destroy path (proc_pgtable_destroy walker, P3-Db) read sub-table contents through the kernel direct map alias. Neither touches TTBR0_EL1 — the walker doesn't depend on the Proc being currently scheduled.

### NEW at P3-Db

131. **Renamed: `vmo_map(Vmo*)` → `vmo_acquire_mapping(Vmo*)`; `vmo_unmap(Vmo*)` → `vmo_release_mapping(Vmo*)`** (P3-Db). The bare refcount-only ops keep their semantics (mapping_count++/-- with the dual-check free transition); only the names changed. The new `vmo_map(Proc*, Vmo*, vaddr, length, prot)` and `vmo_unmap(Proc*, vaddr, length)` are the public API. test_vmo.c was updated; future direct callers of the bare ops should be the VMA layer only.

132. **`vmo_map(Proc*, ...)` rolls back on `vma_insert` overlap rejection** (P3-Db). vma_alloc takes vmo_acquire_mapping (mapping_count++); vma_insert rejection requires vma_free (which fires vmo_release_mapping) to symmetrically revert. The rollback is in `vmo_map` itself, not at the caller — direct vma_alloc/vma_insert callers must mirror this (or use the high-level vmo_map entry).

133. **`vmo_unmap(Proc*, ...)` requires exact-match** (P3-Db). Partial unmap (a sub-range of an existing VMA) is post-v1.0. The v1.0 entry compares VMA `vaddr_start == vaddr` AND `vaddr_end == vaddr+length` and returns -1 on mismatch. Splitting a VMA into two halves with a hole would require allocating a fresh Vma + reordering the sorted list; deferred until userspace `munmap` is wired (Phase 5+).

134. **`proc_pgtable_destroy` walks L0 → L1 → L2 → L3 freeing every reachable sub-table** (P3-Db; closes trip-hazard #116). Leaf descriptors (L3 page entries / L2 block entries) are NOT followed — those user-VA-mapped pages are owned by the VMA layer (VMO mapping_count). The walker uses `pa_to_kva` on each table's PA to read PTE bits via the kernel direct-map alias; it does not touch TTBR0. At v1.0 P3-Db the tree is empty (no PTE installation until P3-Dc demand-paging fault path) so the walker only frees the L0; the recursive form is in place for when sub-tables get populated.

135. **`proc_pgtable_destroy` ordering vs `asid_free`** (P3-Db). `proc.c::proc_free` calls `proc_pgtable_destroy` BEFORE `asid_free`; reversing would also be safe. The Proc is no longer running on any CPU at proc_free time (`wait_pid` spun on `on_cpu == 0`), and `asid_free`'s broadcast `tlbi aside1is` independently covers the ASID's TLB lifecycle BEFORE the ASID is recycled. The walker doesn't issue per-page TLB ops.

### NEW at P3-Bca

108. **DTB still on TTBR0 identity at P3-Bca** — `lib/dtb.c::dtb_init` casts `(uintptr_t)base` instead of `pa_to_kva(base)`. Phase 5+ multi-thread requires a global mmu_lock around demotes (documented).

### NEW at P3-Ba

104. **ASID 0 is RESERVED for kernel TTBR1 mappings** (P3-Ba). User-Proc allocation must use `asid_alloc` which structurally excludes 0. Bypassing the allocator and assigning ASID 0 to a user Proc would conflict with kernel mappings (which use ASID 0 with nG=0).
105. **`asid_free` MUST run before the Proc's struct is reaped** (P3-Ba). The TLB flush in asid_free targets a specific ASID; freeing the Proc without the corresponding asid_free leaks the ASID and leaves stale TLB entries for the next reuser. P3-Bb's `proc_free` wires this in correctly; future hand-coded teardown paths must follow the discipline.
106. **No ASID rollover at v1.0** (P3-Ba). Hard exhaustion at 255 lifetime ASIDs without a single free extincts. v1.0 test scales (≤ ~30 alive Procs at once; cumulative bounded by test count × 2) don't approach this. Phase 5+ adds Linux-style generation rollover.
107. **TLB flush in `asid_free` is INNER-SHAREABLE BROADCAST** (P3-Ba). Cost is multi-cycle and involves all PEs in the IS domain. v1.0 4-vCPU is fine; many-core scaling needs revisit.

### NEW at P3-A

100. **Lock order: `proc_table_lock → r->lock`** (P3-A). Established in `exits()` to keep the wakeup-target Proc alive through the wakeup. The reverse (`r->lock → proc_table_lock`) is **forbidden** — would deadlock against the established order. Verified by code review: `wait_pid_cond` is the only `r->lock`-holder that touches Proc-lineage state, and it deliberately reads without `proc_table_lock` (single-thread-Procs invariant at v1.0).
101. **`wait_pid_cond` lockless walk REQUIRES single-thread Procs** (P3-A). At v1.0 the parent's children list head + sibling chain is single-writer (the parent's own thread). When v1.0 lifts to multi-thread Procs (Phase 5+), `wait_pid_cond` must acquire `proc_table_lock`, AND the sleep protocol must be refactored to break the `r->lock → proc_table_lock` cycle that re-introduces. The refactor likely splits cond evaluation outside `r->lock` with an additional re-check under `r->lock` to preserve atomicity (Linux's pattern for fancy waitqueues).
102. **`g_next_pid` is atomic** (P3-A). `__atomic_fetch_add(&g_next_pid, 1, RELAXED)` — required for SMP cascading rforks where multiple Procs in `proc_alloc` from different CPUs must not collide on the same PID.
103. **Cascading-rfork tests exercise the proc-table-lock paths** (P3-A). `proc.cascading_rfork_wait_smoke` and `proc.cascading_rfork_stress` (100 iters of 3-level lineage) are the regression-protection. The F75 race is timing-dependent; tests don't deterministically trigger it. The lock + spec-level reasoning closes the bug; tests ensure no future change bypasses the locking discipline silently.
